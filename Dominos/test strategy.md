# 1. Introduction

## 1.1 Document Purpose

This document aims to set out testing and supporting processes that will be carried out on as part of the project to implement the KPMG Powered Enterprise solution on Microsoft Dynamics.  

Testing is broken down into a number of phases covered by this document. The Test Strategy covers the Powered Solution and constituent modules and other legacy systems that are subjected to change.

The purpose of the Test Strategy is to ensure that there is a common view of the overall program approach to testing of the new Dynamics 365 Finance and Operations (F&O) solution and its integration to other internal and external systems as part of One Finance One Supply Chain project. The Test Strategy sets out the overall high-level approach and plan for test planning through to test execution and defines the key dependencies, testing deliverables, testing stages and the various types of testing that will be covered.

This document has been developed during the project Validate phase and once approved represents the agreed approach between all testing stakeholders.

## 1.2 Document Scope

The Test Strategy will identify the following for the program

- Powered Test Strategy Design Principles
    
- Testing deliverables
    
- Risks and issues
    
- Testing Types
    
- Test environments
    
- Defect management approach
    
- Testing tools
    
- Test data requirements
    
- Entry/Exit criteria
    
- Test reporting and signoff

## 2.2 Project Test Objectives

The key objectives of Testing in this program are as below:

- Validate through testing that the solution designed and built supports the business processes
    
- Validate through testing that the developments and configuration introduced meet business processes
    
- Validate through testing that the powered solution integrates with legacy systems and the solution functions as expected in an integrated landscape
    
- Validate through testing any new Security roles access operate as per the defined user roles requirements.
    
- Validate through testing that the  Microsoft Dynamics 365 Finance and Supply Chain Solution performs with agreed reliability standards.
    
- Use migrated data to run and validate end to end business processes during User Acceptance Testing
    
- Gain business acceptance of the implemented solution through UAT
    
- Ensure that the solution can transition to live use by validating cutover processes through planned cutover/dress rehearsal before go live.

## 2.5 Test Types In-Scope

Testing on the project will be carried out through the execution of defined Test Phases. Each test phase builds on the previous test phase to collectively provide a comprehensive test approach with each phase targeting specific types of defects.  The test phases control the variables, introducing more complexity gradually to make remediation easier and more efficient saving both time and resources.

Based on the scope of the program / project the following table outlines the standard test phases that are in-scope of this Test Strategy.

|   |   |   |
|---|---|---|
|**Test Types**|**Purpose**|**Owner**|
|Unit Test|Formal testing by the development and the functional team(s) of deviations, including custom code and supporting configuration for D365.|DPE is responsible for testing DPE owned deviations and KPMG is responsible for KPMG owned deviations.<br><br>DPE is responsible for unit testing of the integration components.|
|System Validation Testing (SVT)|Confirming that deviations built or changed are fit for purpose within the constraints of the application and functional areas.|DPE is responsible for testing DPE owned deviations and KPMG is responsible for KPMG owned deviations.|
|System Integration Testing (SIT)|Testing of the integration points of a solution across multiple systems connected via interfaces|DPE|
|End to End Testing (E2E)|Testing of end to end process flow including integration points of a solution across multiple systems connected via interfaces.|DPE|
|User Acceptance Testing (UAT)|Testing of business processes by DPE representatives to give a user’s perspective of the solution validating its suitability for the business.|DPE|
|Regression Testing|Regression testing is re-running functional tests to ensure that previously developed and tested functionality still performs as expected after a change.|DPE|
|Performance Testing|Testing of the performance of a solution against non-functional requirements agreed to by DPE. KPMG will use performance testing tools and specialist techniques to simulate expected load|KPMG|
|Security (Penetration) Testing|Penetration test is an authorized simulated cyberattack on a system, performed to evaluate the security of the system.|DPE|
|Dress Rehearsal (Cutover)|It is a mock-up of the cutover procedure by executing an end-to-end solution with the migrated data.|DPE|
|Production Verification Testing (PVT)|Production Verification Testing refers to testing performed after the Product has been deployed to Production environment before Go Live.|DPE|

## 2.6 Out of Scope

- System Testing on standard Business Processes for Powered Finance
    
- Parallel Test Runs - which means running the old system alongside the new system and comparing the results.
    
- Data Migration Testing is out of scope of KPMG Test Team and documented as part of the Data Migration Strategy Document owned by DPE.
    
- Performance Testing is out of scope for this Test Strategy and documented as part of the Performance Test Strategy scope.
    
- Automation testing for Phase 1 - which is the implementation of an automation tool to execute the test cases and reduce manual regression testing efforts.
    
- Mobile/Device Testing outside of System Validation Testing (SVT) scope.
    

# 3. Powered Enterprise Test Methodology

## 3.1 Test Approach

KPMG Powered Enterprise™ an incrementally phased approach, aimed at reducing the time and risk associated with transformation activities. To achieve this, this test methodology is integral with every phase. The Construct Phase is targeted to run in an iterative manner with System Testing, System Integration Testing before the solution is deployed for User Acceptance Testing in the Deploy phase.

## 3.2 Testing Types

Different Testing types are applicable throughout the lifecycle of the Project. The below section provides the detailed approach to each of the test type as in scope of the Project One Finance One Supply Chain.

### 3.2.1 Unit Testing

Unit Testing is a level of software testing where individual units/ components of a software are tested. The purpose is to validate that each unit of the software performs as designed. A unit is the smallest testable part of any software. It usually has one or a few inputs and usually a single output.

This type of testing is done by the KPMG Developer/Consultant of the User Story/module/unit. There will not be formal test scripts written for unit testing in the testing tool. The testing will be done in the Dev Environment.

### 3.2.2 System Validation Testing (SVT)

The purpose of System Validation Testing (SVT) is to evaluate the system’s compliance with the specified User Stories acceptance criteria/deviation specification. System testing is performed during the duration of the Construct Phase by KPMG Test Team in a Test Environment.

The requirements will be captured in Jira and the System test cases will be build based on the respective user stories’ acceptance criteria and documented in the selected testing tool of Zypher scale.

The test cases will include the positive and negative testing scenarios within the scope of the respective user story/deviation. The system testing will be done against the configuration and any deviation requirements. If the user story defines a particular new security role, then the system testing is conducted based on the respective defined Security Roles.

For Powered Microsoft Finance, the System Testing scope include testing the deviation user stories once they are build, unit tested and deployed into the test environments progressively.

For Powered Finance, the standard Business processes that have not been changed are already tested to some degree as part of the shakeout completed by the Functional Team once the Gold Environment is set up and loaded with DPE related Configuration data. There is no specific system testing required on the Powered Finance Standard Business Processes.

### 3.2.3 System Integration Testing (SIT)

System Integration Testing (SIT) is defined as a type of formal testing of the integration points of a solution where the integration is across functional areas of multiple systems connected via interfaces. The purpose of this level of testing is to expose defects in the interaction between these software solutions when they are integrated and will be tested by DPE.

Point to point System Integration Testing can be done during the Construct phase when the integration build has been completed and the build is unit tested and deployed in the SIT Environment. In the end of Construct phase a dedicated phase is run for System Integration Testing before it can be passed onto for User Acceptance Testing.

The scope of System Integration Testing  at the end of Construct phase is to exercise as many Business processes that have deviations and integrations with focus on critical business scenarios.

### 3.2.4 End-to-End Testing (E2E)

The purpose of End-to-End Testing (E2E) is to test the end to end process flow including integration points of a solution across multiple systems connected via interfaces. It verifies that each component functions correctly within the entire system in real-world scenarios, ensuring the system behaves as per the defined requirements.

This phase will be executed after SIT and will be owned by DPE.

### 3.2.5 User Acceptance Testing (UAT)

The purpose of User Acceptance Testing (UAT) is to ensure that the developed solution meets the needs of One Finance One Supply Chain as defined in the approved requirements. When UAT is signed off as having met the exit criteria for the solution and migrated data, DPE is deemed to have accepted the solution and migrated data.

UAT will be performed after E2E testing and managed by DPE. The business SMEs/ UAT Lead from DPE will be responsible for the planning and co-ordination of resourcing, test design, test preparation, test environments readiness, data refresh in UAT Environment, test execution, Defect Management, and reporting. The UAT Lead will ensure the test coverage for UAT include the end-to-end business operational scenarios and validating the migrated data is correct.

UAT scenarios are based on user stories and requirement specification with a key outcome being validating that users of the system can execute real-world scenarios. This will include end to end testing of standard business processes and processes with deviation and integrations.

UAT will be performed using migrated data, as opposed to data created for the purposes of testing. If UAT identifies a data migration or other data quality-related issue, data will be created to complete testing if required.

KPMG will support DPE during UAT by providing support on UAT defect fixes.

### 3.2.6 Regression Testing

Regression testing is a type of testing that is done to verify that a code change does not impact the existing working functionality build earlier. This is to make sure the solution continues to operate correctly after new functionality or bug fixes have been applied. Regression testing is performed by repeating select test cases, known collectively as the Regression Suite and will be tested by DPE.

The Regression Suite is accumulated over time as critical test cases built during System testing and System integration testing will be planned to re-run at different stages of testing, or after significant changes are made in the test environments or across different releases of the project.

### 3.2.7 Performance Testing

The Performance testing will be done by KPMG Environment Team as per the defined Non-Functional Requirements. For details, refer to the Performance Testing strategy.

# 4. Test Standards

## 4.1 Test tools

The Powered Execution Suite is a tightly integrated collection of tools that supports the delivery of Powered Solutions. Testing is supported by Zephyr scale and Jira. Zephyr Scale is used to manage the creation and execution of test scripts, including documenting the results and associated evidence. Jira is integrated with Zephyr and is used to manage the defect resolution process. In addition to managing the defect process, Jira is used to provide progress dashboards to monitor execution and defect statistics in real time. Non-functional Testing will make use of specific tools and techniques. The tools selected to support each NFT type will be documented in the respective Test Strategy document. The Zephyr scale license and procurement will be managed by DPE.

*Decision pending on the test management tool Zephyr Scale at the time of drafting this Test Strategy and will be finalised at the end of the Validate phase.

## 4.2 Test Scoping

Test Scoping for System testing will take place as part of Construct phase and will be done by the KPMG Functional or Test Management Team to meet the Functional requirements of the new system.

Test Scoping for SIT and End to End testing will takes place as part of Construct phase and will be done by the Client Test Manager. The DPE Test Management Team will prepare a proposed scope by reviewing the Integration User stories, Integration Specifications, end to end business scenarios and then present these to Functional SMEs and Business Process Owners at Test scoping workshops. During these sessions the test scenarios will be refined by the business who also provide business criticality scores, which are then used to prioritise the tests. The outcome of the test scoping workshops is an agreed list of prioritised integration test scenarios which match the processes used to operate the business post go-live.

UAT Scoping is performed by the Client Test Manager during the construct phases. The scope for UAT will be based on the documented and signed off requirements most likely in the form of user stories in Jira. Business SMEs will have the knowledge to scope the operational scenarios for UAT in addition to the documented and signed off requirements.  Business process flows are for reference only and will not form a formal test basis document.

For Performance testing, a workload model will be used to capture transactional volumes both at historic and projected levels. This will then be used to determine which processes should be prioritised for performance testing. This will be managed by the Environment Team.

For other types of NFT, tests will be designed to meet one or more of the non-functional requirements

## 4.3 Test Scripts

The test scripts describe not only the high-level scenario of a process, but it also describes each action that requires execution and the expected result from the action described. These scripts will provide the specific data to use in each event and will describe how and where the data should be used.

Wherever possible, Test Scenarios and Test scripts will be written to be elaborated and re-used on subsequent test phases and to be useable during BAU.

For System Testing, the Functional or Testing team will create the test scripts which will be reviewed by the nominated Functional or Test Lead prior to test execution. For SIT and End to End testing, the test scenarios will be created by DPE nominated representatives from the business and are required to review and sign off the Test Scenarios prior to test execution.

For UAT test phases nominated representatives from the business are required to create, review, and sign off the Test Scenarios prior to test execution and ensure the UAT Test Scripts include the business scenarios and confirm traceability to the scope.

## 4.4 Traceability

Powered is a process-based approach and functional tests are defined and developed from the defined process models, the Business process flows act as the requirements of the solution and the tests coverage is mapped back to the process flows and the agreed deviation requirements captured as user stories. The requirements will be tracked in Jira as user stories and test cases in Zypher scale will be mapped to the requirements.

Non-functional tests will be traced back to the Non-functional requirements documented as part of the Validate Phase.

## 4.5 Test Execution

Test Cycles will be set up by Test Phase within Zephyr Scale. Cycles will contain the Test Scenarios for each Workstream. Test Scripts will be executed in the Zephyr Scale, recording the actual result for each step with their relevant evidence. This is required to facilitate the auditing process where applicable. Document numbers generated as part of the test (for example document numbers, voucher numbers, record Ids etc.) must be recorded in the “Actual Result” field during test execution.

Evidence in the form of a screenshot showing the objective of a key test case being achieved will be captured in Zephyr Scale and stored against the relevant test case. A key step is defined as:

- A step that generates a document or reference number
    

- A step where an approval (or rejection) is made
    

- A step where the objective of the test is reached.
    

Any screenshot evidence recorded during test execution on the Production Data in Environment should mask for any sensitive/identifiable data by the testers/developers/users when attaching to the test Results or Defects. If the tester has no specific requirement for the test scenario to use migrated data than they should preferably create their own data and run the test to capture the test evidence without masking.

A test script is considered to have “passed” if the objective of the script has been achieved. Conversely, a test script is considered “failed” if the objective is not accomplished.

# 7. Defect Management

## 7.1 Defect Management Overview

This section describes the process by which defects will be captured and monitored during their lifecycle, from creation to closure. A defect is defined as any failure that prevents testing being undertaken or completed. Defects will be raised during testing for any error that occurs that impacts the ability to test the system, business process and/or transaction.

All defects will be logged in Jira, the program defect management tool where they will be linked to the tests that they impact and reviewed and tracked until they are resolved. All defects will be prioritised based on their impact on the business should the fault occur in live (severity) and the impact on testing’s ability to progress (priority).

A defect is considered resolved when it is in one of the statuses described below:

“Closed – Rejected”– It has been demonstrated that the process or system is operating as designed and agreed this with the test analyst(s) who reported the defect.

“Closed” – A fix has been applied and where necessary the change has been promoted to the test environment, and the tester has demonstrated that the process or system is now operating as designed.

“Change Request” – This status is set once there is agreement that the defect relates to a new requirement rather than the solution working as designed.

## 7.2 Defect Resolution Process

Any defects found during testing activities will be raised in Jira, the defect management tool for the program. All defects must contain a meaningful title, a description of the fault, re-creation steps, screenshots of the error (masked if containing production sensitive data) and other evidence such as log files. When creating a defect, the Test Analyst must assess the business impact (severity) and the impact on ability to progress testing (priority) using the guidance in this document. If a critical or urgent defect is detected, Test Analysts are expected to raise this as a priority with Test Manager to ensure that they can prioritise resolution to enable efficient execution.

Statuses within Jira are used to manage the resolution process, when creating a defect Test Analysts create the defect in status “New” and assign this to the defect manager for triage. Once the defect is accepted, the defect manager will assign the issue to the appropriate resource for resolution. The Defect Process Flow diagram in section 7.2.1 contains details of the statuses used to manage the defect resolution.        

When raising or amending a defect, for example updating the status or assignment, appropriate comments must be logged to create a record of the actions taken. This helps to provide sufficient information to the resource assigned to resolve the issue and enables the tester to monitor progress of the defect and understand the steps they need to take to retest.Defect Process flow.docx

Data on the count and status of defects can be made available on the report issued during test execution at an agreed frequency during Construct Phase.

If any defect is found in the out of box product, these will not be fixed prior to phase exit and/or go-live. Either the defect is accepted and closed off or workarounds will be developed and tested as part of the scope of the program. Workarounds must be accepted by the appropriate DPE representative prior to implementation and testing. The retesting of Product defect resolutions and the decommissioning of workarounds post go-live would be the responsibility of DPE.

## 7.3 Defect Severity Ratings

Defect severity is used to describe the business impact of a defect should the fault occur in live. All defects must be assigned a severity by the Test Analyst when creating the defect, this will then be reviewed and confirmed at the triage meeting.

|   |   |
|---|---|
|**Severity**|**Description**|
|1 – Critical|A defect, which has a severe impact, with no workaround available, including:<br><br>- System crashes or does not operate at all<br>    <br>- Major items of functionality covered by the Specification are unavailable (prevents key business requirements being met)<br>    <br>- Prevents a critical business process from being completed<br>    <br>- Impacts majority of users within critical processes|
|2 – High|A defect, which has a major impact for which there is a short-term temporary workaround. Examples include:<br><br>- Items of functionality included in the Specification unavailable<br>    <br>- Prevents a critical business process from being completed|
|3 – Medium|A defect, which has a minor impact whereby the system is operable, and an acceptable workaround is available.<br><br>Non-critical steps of the business process are impaired but work around exists.<br><br>- Impacts a limited number of users of the relevant specified functionality<br>    <br>- Minor items of specified functionality not available|
|4 – Low|Other defects, addressable by KPMG, which are cosmetic in nature, for which no workaround is required. Examples include: -<br><br>- Minor cosmetic issues<br>    <br>- Those issues that do not impact completion of any business processes as specified|

## 7.4 Defect Priority Ratings

Defect priority relates to the impact on testing and its ability to progress at the planned rate. All defects must be assigned a priority by the Test Analyst when creating the defect, this will then be reviewed and confirmed at the triage meeting.

|   |   |
|---|---|
|**Priority**|**Definition & Impact on Testing**|
|1 – Urgent|All testing in a functional area is blocked.<br><br>Team member(s) unable to progress any tests they are responsible for.|
|2 – High|Significant numbers/proportion of tests within a functional area are impacted.<br><br>Team member(s) can continue testing but significant re-test will be required upon fix.|
|3 – Medium|Defect is blocking a small number/proportion of tests.<br><br>Team member(s) can continue with the test|
|4 – Low|Able to complete the test with minor deviation from expected results (i.e., minor appearance/usability issues). Defects at this level may be deferred or not fixed at all.|

## 7.5 Defect Management Meetings

Throughout SIT, E2E and UAT test execution, a regular Defect Triage meeting will be held, chaired by the Test Lead, and attended by representatives of the teams responsible for fixing and closing the defects. The meeting has the following aims:

- To validate those issues detected are genuine defects and not change requests
    
- To validate that the guidance on Severity and Priority have been properly interpreted by the Test Analyst
    
- To review the progress of all Critical and High severity issues, ensuring daily progress to completion
    
- To review the progress on items due for delivery and confirm that they are on track for release into test environment
    
- To review the items pending retest and to confirm when/if they can be closed
    

The following participants are required to attend

- KPMG Test Lead
    
- DPE Test Lead (Chair and meeting organiser for SIT, E2E and UAT)
    
- Functional Workstream Leads
    
- Environment Manager
    
- Data Migration Lead
    
- Business SMEs
    
- Technical Leads for Workstream
    
- Technical Leads from DPE for integrating applications (if applicable)
