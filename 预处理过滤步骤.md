具体做法通常不是一句“过滤掉”，而是做成一条**预处理流水线**：

**原始文档 → 切成较小单元 → 审核打分 → 按规则处理 → 再进入 chunking / embedding / 索引。**

这样做比“整篇文档直接删掉”更稳，因为冒犯性语言往往只出现在某几段、某几句里。OpenAI 的 moderation endpoint 可以用来判断文本是否可能有害；Azure AI Content Safety 则支持对单句或整个数据集做文本审核、调灵敏度，并支持自定义 blocklist。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation "Moderation | OpenAI API"))

## 一般怎么落地

### 1. 先把文档切成可审核的最小单位

不要一上来就对整本 PDF、整篇网页做“删/留”判断。更常见的是先按这些粒度切：

- 段落
    
- 句子
    
- 页面
    
- 预切好的 chunk
    

原因很简单：  
如果第 7 段有辱骂词，没必要把前 6 段和后 10 段一起删掉。

### 2. 先做一个便宜的初筛

通常先用**关键词 / blocklist / 正则**做第一层筛查，把明显脏词、辱骂词、常见变体先标出来。Azure Content Safety 明确支持 blocklist management，这类 blocklist 很适合放企业自定义禁词。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/overview "What is Azure AI Content Safety? - Azure AI services | Microsoft Learn"))

这一步的目的不是最终裁决，而是：

- 降低后面模型审核成本
    
- 把高风险片段先挑出来
    
- 提高召回率
    

### 3. 再用审核模型做二次判定

然后把被初筛命中的片段，或者全部片段，送到审核模型里打分类和严重度。

可选做法有两种：

- **OpenAI moderation**：适合做 harmful content 分类，OpenAI 官方说可以用 moderations endpoint 检查文本或图片是否可能有害。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation "Moderation | OpenAI API"))
    
- **Azure AI Content Safety**：适合需要分门别类和设阈值的场景，Azure 会给出类别和 severity，类别包括 `Hate`、`Sexual`、`Violence`、`SelfHarm`，并支持 0–7 严重度。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories "Harm categories in Azure AI Content Safety - Azure AI services | Microsoft Learn"))
    

### 4. 不同分数做不同动作

真正可用的系统，通常不是简单“命中就删除”，而是按阈值分流：

- **低风险**：保留，但打标签
    
- **中风险**：替换脏词、遮盖敏感词、或丢弃该段
    
- **高风险**：直接隔离，不进入索引
    
- **不确定 / 边缘样本**：人工复核
    

这是最关键的地方。

因为 Azure 明确说明，有些词虽然看起来敏感，但可能出现在**教育、媒体、历史、医学、科学**等安全语境里，这类内容不应该一刀切删掉。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories "Harm categories in Azure AI Content Safety - Azure AI services | Microsoft Learn"))

## 一个最实用的规则例子

你可以先定一个企业规则：

- `Hate >= 2`：删掉该段
    
- `Sexual >= 4`：删掉该段
    
- `Violence >= 4`：删掉该段
    
- `SelfHarm >= 2`：删掉该段
    
- 仅包含轻微粗口但无仇恨指向：替换成 `[OFFENSIVE_TERM]`
    
- 命中教育/新闻/法律上下文：保留但打标签，供后续检索时降权
    

这个规则不是官方固定规则，而是基于 Azure 的类别和严重度能力做出来的业务策略。Azure 官方明确提供类别和 severity，用于配置 moderation 和 compliance。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories "Harm categories in Azure AI Content Safety - Azure AI services | Microsoft Learn"))

## 一个 Python 版最小流程

下面这个例子演示的是：

- 先按段落切
    
- 先跑 blocklist
    
- 再跑审核
    
- 最后决定保留 / 遮盖 / 删除
    

```python
import re

BAD_WORDS = {"idiot", "stupid", "slur1", "slur2"}

def split_paragraphs(text: str) -> list[str]:
    return [p.strip() for p in text.split("\n\n") if p.strip()]

def blocklist_hit(text: str) -> bool:
    lower = text.lower()
    return any(w in lower for w in BAD_WORDS)

def mask_offensive_terms(text: str) -> str:
    out = text
    for w in BAD_WORDS:
        out = re.sub(rf"\b{re.escape(w)}\b", "[OFFENSIVE_TERM]", out, flags=re.IGNORECASE)
    return out

def moderate_text_with_provider(text: str) -> dict:
    """
    这里替换成真实的 OpenAI moderation 或 Azure Content Safety 调用。
    返回示例:
    {
        "hate": 4,
        "sexual": 0,
        "violence": 0,
        "self_harm": 0
    }
    """
    return {"hate": 0, "sexual": 0, "violence": 0, "self_harm": 0}

def decide_action(scores: dict, block_hit: bool) -> str:
    if scores["hate"] >= 2:
        return "drop"
    if scores["sexual"] >= 4:
        return "drop"
    if scores["violence"] >= 4:
        return "drop"
    if scores["self_harm"] >= 2:
        return "drop"
    if block_hit:
        return "mask"
    return "keep"

def clean_document(text: str) -> list[str]:
    kept = []
    for para in split_paragraphs(text):
        hit = blocklist_hit(para)
        scores = moderate_text_with_provider(para)
        action = decide_action(scores, hit)

        if action == "drop":
            continue
        elif action == "mask":
            kept.append(mask_offensive_terms(para))
        else:
            kept.append(para)

    return kept
```

这段代码里最重要的是 `decide_action()`。  
也就是：**审核结果不是只拿来报警，而是拿来决定这段文本还能不能进向量库。**

## 如果你用 OpenAI moderation，怎么接

OpenAI 官方给了最直接的写法：

```python
from openai import OpenAI

client = OpenAI()

response = client.moderations.create(
    model="omni-moderation-latest",
    input="...text to classify goes here..."
)
```

OpenAI 明确说明 moderations endpoint 用来检查文本或图片是否可能有害，并可据此采取过滤等行动。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation "Moderation | OpenAI API"))

所以你的 `moderate_text_with_provider()` 可以换成类似：

```python
def moderate_text_with_provider(text: str) -> dict:
    response = client.moderations.create(
        model="omni-moderation-latest",
        input=text
    )
    result = response.results[0]
    # 按你自己的策略把返回结构映射成统一分数字段
    return {
        "hate": int(bool(result.categories.hate)),
        "sexual": int(bool(result.categories.sexual)),
        "violence": int(bool(result.categories.violence)),
        "self_harm": int(bool(result.categories.self_harm)),
    }
```

OpenAI 这一套更适合做“有害/无害”判断和快速过滤。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation "Moderation | OpenAI API"))

Azure 更适合做“按类别 + 严重度阈值”的策略。  
它支持：

- 单句或整数据集测试

它给出的类别和严重度对这件事很有用，因为你可以很明确地写规则：

Azure 文档还特别提醒：有些看起来敏感的内容，在教育、历史、媒体、医学等语境下可以是安全内容。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories "Harm categories in Azure AI Content Safety - Azure AI services | Microsoft Learn"))

## 真正上线时，最推荐的策略

最稳的不是“只删”，而是这四种动作并存：

### 1. Drop

特别明确的冒犯性内容，直接不进索引。

### 2. Mask

脏词替换掉，但保留可用语义。

例如：

- `"you are an idiot"`  
    变成
    
- `"you are an [OFFENSIVE_TERM]"`
    

这样还能保留上下文，而不会把整段知识都丢掉。

### 3. Tag

给 chunk 打标签：

- `contains_offensive_language=true`
    
- `moderation_hate_score=2`
    

以后检索时可以：

- 降权
    
- 只在某些场景允许召回
    
- 做审计
    

### 4. Human review

边界样本进人工复核队列。

## 在 RAG 里应该放在哪一步

最推荐顺序是：

**原始文档 → 基础清洗 → 冒犯性语言过滤/遮盖 → chunking → embedding → vector index**

不要等到模型回答时再处理。  
因为如果你先 embedding 再过滤，脏内容可能已经：

- 进了向量库
    
- 参与召回
    
- 出现在 prompt 里
    

Databricks 的 AI Gateway / Guardrails 更适合做**在线请求/响应阶段**的保护，比如对请求和响应做内容审核、PII 检测和 dry run；它更像第二道保险，不是离线构建 RAG 数据集时的主要清洗方式。([Databricks Documentation](https://docs.databricks.com/aws/en/ai-gateway/configure-endpoints-beta "Configure AI Gateway endpoints | Databricks on AWS"))

## 一句最实用的总结

**对 RAG 数据集中的冒犯性语言，最稳的做法是：先按段落或 chunk 切开，用 blocklist 做初筛，再用审核模型按类别和严重度打分，然后对不同风险级别执行 keep / mask / drop / review，最后再进入 embedding 和索引。** OpenAI moderation 适合快速有害内容分类，Azure Content Safety 适合按类别和 severity 配规则，并支持 blocklist 和整数据集调试。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation "Moderation | OpenAI API"))
