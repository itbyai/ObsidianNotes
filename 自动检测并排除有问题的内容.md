可以把它理解成一条 **“先分类，再决策”** 的清洗流水线。

**文本分类模型**在这里不是拿来回答问题，而是拿来给内容打标签：这段文本是不是包含仇恨、性内容、暴力、自残、辱骂、敏感话题，或者是否达到某个风险等级。OpenAI 的 moderation endpoint 用来识别“可能有害”的文本或图片；Azure AI Content Safety 会按 **hate、sexual、violence、self-harm** 等类别给出严重度；Google Cloud 的文本审核也会返回 harmful/sensitive categories。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation?utm_source=chatgpt.com "Moderation | OpenAI API"))

真正落地时，通常不是“整篇文档扔进模型，返回一个 yes/no”，而是下面这 6 步：

**1. 先切分。**  
把原始文档先切成段落、句子或预备 chunk。这样做的原因是：有问题的内容常常只在局部，没必要整份文档一起删掉。Azure Content Safety 和 Google 的文本审核都支持针对输入文本做分类与评分，因此很适合对较小文本单元逐段处理。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/overview?utm_source=chatgpt.com "What is Azure AI Content Safety?"))

**2. 跑分类/审核模型。**  
把每个段落送进审核模型，拿到类别和分数。  
常见输出会类似：

- hate = 低/中/高
    
- sexual = 低/中/高
    
- violence = 低/中/高
    
- self-harm = 低/中/高
    

Azure 明确支持按类别和 severity 返回结果；OpenAI moderation 也会返回各类有害内容的分类结果；Google 的 moderateText 也会返回 moderation categories。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories?utm_source=chatgpt.com "Harm categories in Azure AI Content Safety"))

**3. 设规则阈值。**  
分类结果出来后，不是直接全删，而是用规则引擎决定怎么处理。  
例如：

- `hate >= 中等` → 丢弃该段
    
- `violence = 低` → 保留但打标签
    
- `sexual = 高` → 丢弃
    
- `辱骂但不是仇恨言论` → 只遮盖脏词
    
- `边界样本` → 人工复核
    

Azure 的分类体系本身就是为这种“按类别和严重度做策略”设计的。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories?utm_source=chatgpt.com "Harm categories in Azure AI Content Safety"))

**4. 按动作处理。**  
常见动作有 4 种：

- **keep**：保留
    
- **mask**：遮盖部分词
    
- **drop**：删除句子/段落
    
- **review**：进入人工复核
    

这一步很关键，因为“检测”只是识别风险，“排除”才是把高风险内容挡在向量库和模型外面。OpenAI 官方也明确说，识别到有害内容后可以采取 filtering 或其他 corrective action。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation?utm_source=chatgpt.com "Moderation | OpenAI API"))

**5. 再进入 RAG 数据管道。**  
只有处理后的干净文本才进入后续：

`清洗后文本 → chunking → embedding → vector index → retrieval → LLM`

这样可以避免有问题的内容先进入 embedding 和向量库，后面又被召回给模型。Databricks/任何 RAG 架构里，这一步都应该放在索引前。这个结论是工程实践推导，不是某一家产品的专属要求。

**6. 最后再做一层输出审核。**  
即使输入语料清洗过，模型输出时仍可能生成不合适内容，所以很多系统还会在 **response 侧再跑一次审核模型**。OpenAI moderation 和 Azure Content Safety 都适合做输入输出双向筛查。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation?utm_source=chatgpt.com "Moderation | OpenAI API"))

## 一个最实用的例子

假设你的 RAG 知识库里有客服文本：

> “那个用户就是个白痴，别理他。”  
> “客户威胁说要伤害同事。”  
> “标准退款流程如下……”

分类模型跑完后可能得到：

- 第一句：辱骂 / hate 低，violence 无
    
- 第二句：violence 中
    
- 第三句：安全
    

然后规则可以这样写：

- 辱骂但非核心知识 → `mask`
    
- 暴力威胁 → `drop`
    
- 安全文本 → `keep`
    

处理结果可能变成：

- “那个用户就是个 `[OFFENSIVE_TERM]`，别理他。”
    
- 第二句整段删除
    
- 第三句原样保留
    

这样做的好处是：**既减少风险，又尽量保留业务知识。**

## 常见实现方式

**方案 A：用现成审核 API**  
最省事。直接调用 OpenAI moderation、Azure Content Safety 或 Google 的 text moderation，拿类别和分数，再自己写规则。OpenAI moderation endpoint 是官方提供的内容审核接口；Azure AI Content Safety 是专门做 harmful content 检测的；Google 也提供 moderateText。([OpenAI Developers](https://developers.openai.com/api/docs/guides/moderation?utm_source=chatgpt.com "Moderation | OpenAI API"))

**方案 B：自己训练文本分类模型**  
如果你的“问题内容”定义很业务化，比如：

- “脏话投诉”
    
- “恶意工单”
    
- “法律风险句子”
    
- “竞争对手攻击”
    

那就可以自己标注数据，训练一个多标签文本分类器。  
常见做法是把每段文本标成：

- safe
    
- offensive
    
- abusive
    
- threatening
    
- sensitive
    

然后让模型预测标签和置信度，再用阈值决定 keep/drop。这个属于通用机器学习方法，不依赖某个具体平台。

## 为什么要用文本分类模型，而不是只靠关键词

因为关键词方法很脆弱。

比如：

- 变体拼写
    
- 隐晦表达
    
- 上下文不同
    
- 教育/新闻/医学语境
    

Azure 官方就明确提醒，某些敏感词在教育、历史、医学等上下文里可能是安全内容，因此要结合语境和严重度，而不是简单按词过滤。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/harm-categories?utm_source=chatgpt.com "Harm categories in Azure AI Content Safety"))

所以最好的实践通常是：

**关键词 / blocklist 做初筛，文本分类模型做正式判断。**

**“使用文本分类模型自动检测并排除有问题的内容”**，本质上就是：
