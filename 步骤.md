可以，最常见、最实用的做法就是：

**在进入模型之前，先做静态脱敏**。  
也就是把原始文本里的邮箱、手机号先替换掉，再送去做 embedding、RAG、LLM 推理。

这样比“只靠 prompt 让模型别说出来”更稳。

---

## 1. 最简单的思路

原文：

```text
请联系张三，手机号 0412 345 678，邮箱 zhangsan@example.com
```

脱敏后：

```text
请联系张三，手机号 [PHONE]，邮箱 [EMAIL]
```

或者部分遮盖：

```text
请联系张三，手机号 0412***678，邮箱 z***@example.com
```

---

## 2. Python 里怎么做

这是最直接的版本。

```python
import re

EMAIL_PATTERN = r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b'
PHONE_PATTERN = r'(?:(?:\+?\d{1,3}[-.\s]?)?(?:\(?\d{2,4}\)?[-.\s]?)?\d{3,4}[-.\s]?\d{3,4})'

def mask_email(text: str) -> str:
    return re.sub(EMAIL_PATTERN, "[EMAIL]", text)

def mask_phone(text: str) -> str:
    return re.sub(PHONE_PATTERN, "[PHONE]", text)

def mask_sensitive(text: str) -> str:
    text = mask_email(text)
    text = mask_phone(text)
    return text

sample = "请联系张三，手机号 0412 345 678，邮箱 zhangsan@example.com"
print(mask_sensitive(sample))
```

输出可能是：

```text
请联系张三，手机号 [PHONE]，邮箱 [EMAIL]
```

## 3. 如果你想“部分保留”，而不是全替换

### 邮箱部分遮盖

```python
import re

EMAIL_PATTERN = r'\b([A-Za-z0-9._%+-])([A-Za-z0-9._%+-]*)(@[A-Za-z0-9.-]+\.[A-Za-z]{2,})\b'

def partial_mask_email(text: str) -> str:
    return re.sub(EMAIL_PATTERN, r'\1***\3', text)

print(partial_mask_email("zhangsan@example.com"))
```

输出：

```text
z***@example.com
```

### 电话部分遮盖

```python
import re

def partial_mask_phone(text: str) -> str:
    def repl(match):
        raw = match.group(0)
        digits = re.sub(r'\D', '', raw)
        if len(digits) < 6:
            return "[PHONE]"
        return digits[:4] + "***" + digits[-3:]
    pattern = r'(?:(?:\+?\d{1,3}[-.\s]?)?(?:\(?\d{2,4}\)?[-.\s]?)?\d{3,4}[-.\s]?\d{3,4})'
    return re.sub(pattern, repl, text)

print(partial_mask_phone("0412 345 678"))
```

输出：

```text
0412***678
```

## 4. 在 PySpark / Databricks 里怎么做

如果你在 Databricks 里处理表数据，这样更常见。

```python
from pyspark.sql import functions as F

df = spark.createDataFrame([
    ("请联系张三，手机号 0412 345 678，邮箱 zhangsan@example.com",),
], ["text"])

email_pattern = r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b'
phone_pattern = r'(?:(?:\+?\d{1,3}[-.\s]?)?(?:\(?\d{2,4}\)?[-.\s]?)?\d{3,4}[-.\s]?\d{3,4})'

masked_df = (
    df
    .withColumn("masked_text", F.regexp_replace("text", email_pattern, "[EMAIL]"))
    .withColumn("masked_text", F.regexp_replace("masked_text", phone_pattern, "[PHONE]"))
)

display(masked_df)
```

## 5. 在 SQL 里怎么做

如果你想直接在 Databricks SQL 里处理：

```sql
SELECT
  text,
  regexp_replace(
    regexp_replace(
      text,
      '\\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}\\b',
      '[EMAIL]'
    ),
    '(?:(?:\\+?\\d{1,3}[-.\\s]?)?(?:\\(?\\d{2,4}\\)?[-.\\s]?)?\\d{3,4}[-.\\s]?\\d{3,4})',
    '[PHONE]'
  ) AS masked_text
FROM my_table;
```

## 6. 放到 RAG / LLM 流程里，应该放在哪

最推荐顺序：

### 方案 A：进入模型前先脱敏

`原始文本 -> 脱敏 -> chunking -> embedding -> vector search -> LLM`

这是最稳的。

因为：

- 向量库里不会存原始敏感信息
    
- prompt 里也不会带原始敏感信息
    
- 模型更不容易泄露
    

### 方案 B：输出后再做一次过滤

`LLM 输出 -> 再扫描一次 -> 遮盖敏感字段`

这是第二道保险。

## 7. 更完整的工程做法

实际项目里通常有 3 种级别：

### 第一种：纯替换

- `[EMAIL]`
    
- `[PHONE]`
    

优点：最简单最安全  
缺点：丢失部分信息

### 第二种：部分保留

- `z***@example.com`
    
- `0412***678`
    

优点：还能看出大概是谁/哪个域名/哪个号码段  
缺点：仍然保留了一点敏感信息

### 第三种：令牌化

把真实值替换成：

- `[EMAIL_001]`
    
- `[PHONE_001]`
    

并把映射单独存到安全表里：

|token|real_value|
|---|---|
|EMAIL_001|[zhangsan@example.com](mailto:zhangsan@example.com)|
|PHONE_001|0412345678|

优点：可追溯、可恢复  
缺点：实现更复杂，需要权限控制

## 8. 常见坑

### 正则写太宽

会把订单号、邮编、ID 误识别成电话。

### 正则写太窄

像国际号码、带括号号码、带空格号码匹配不到。

### 只做输入不做输出

模型可能根据上下文又“复述”出来，所以输出最好也扫一次。

### 只靠 prompt

比如写“不要输出敏感信息”，这不够稳。  
真正安全的是**先静态脱敏，再让模型处理**。

**最稳的做法：先用正则或 PII 检测把邮箱、电话静态脱敏，再送进向量库和模型。**
