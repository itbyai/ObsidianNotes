你创建了 API 之后，**本质上是给模型做了一个“统一入口”**。  
以后谁想用这个模型，不需要再进 Databricks 找 notebook，也不需要自己加载模型文件，**只要调用这个 endpoint 就行**。Databricks 官方把 Model Serving 定位为用于 **real-time serving** 和 **batch inference** 的部署方式；custom model endpoint 通过 REST 接口接收请求，Foundation Model endpoint 也可以用 REST 方式调用。([Databricks Documentation](https://docs.databricks.com/aws/en/machine-learning/model-serving/?utm_source=chatgpt.com "Deploy models using Mosaic AI Model Serving"))

最常见的使用位置有这几类：

**1. 业务系统后端**  
比如你的网站、App 后台、企业内部系统。  
用户点了一个按钮，后端代码就去调这个 API，拿回预测结果再返回给前端。  
这类最典型，比如：

- 情感分类
    
- 风险评分
    
- 文本总结
    
- RAG 问答
    
- embedding 生成  
    Databricks 官方说明，custom model endpoint 和 foundation model endpoint 都是通过服务端请求来调用的。([Databricks Documentation](https://docs.databricks.com/aws/en/machine-learning/model-serving/score-custom-model-endpoints?utm_source=chatgpt.com "Query serving endpoints for custom models"))
    

**2. 前端应用背后的服务层**  
通常不是浏览器直接调，而是前端调用你自己的后端，你自己的后端再去调 Databricks endpoint。  
原因是 token、权限、限流、错误处理一般不适合直接暴露到前端。Databricks 的调用方式本身是基于受保护的 REST API，需要认证信息。([Databricks Documentation](https://docs.databricks.com/aws/en/machine-learning/model-serving/score-custom-model-endpoints?utm_source=chatgpt.com "Query serving endpoints for custom models"))

**3. Databricks Notebook / Job 里**  
即使你在 Databricks 内部，也可以把 endpoint 当成一个在线服务来调用。  
这适合：

- 测试线上模型
    
- 跑批量数据
    
- 做回归验证
    
- 在工作流里串联调用  
    Databricks 官方说明，Model Serving 既支持实时调用，也支持批量推理场景。([Databricks Documentation](https://docs.databricks.com/aws/en/machine-learning/model-serving/?utm_source=chatgpt.com "Deploy models using Mosaic AI Model Serving"))
    

**4. Databricks SQL 里**  
如果你想在 SQL 里直接调用模型，可以用 `ai_query()`。  
Databricks 官方说明，`ai_query` 可以调用已有的 Model Serving endpoint，用于 **real-time inference** 或 **batch inference**，也能查询部署在 serving endpoint 上的 agent。([Databricks Documentation](https://docs.databricks.com/aws/en/sql/language-manual/functions/ai_query?utm_source=chatgpt.com "ai_query function | Databricks on AWS"))

例如概念上就是这种用法：

```sql
SELECT ai_query('your-endpoint-name', question)
FROM your_table;
```

这很适合：

- 批量补充标签
    
- SQL 分析时直接调模型
    
- 给表格数据做离线 enrichment ([Databricks Documentation](https://docs.databricks.com/aws/en/sql/language-manual/functions/ai_query?utm_source=chatgpt.com "ai_query function | Databricks on AWS"))
    

**5. 外部自动化流程 / 第三方系统**  
比如：

- Airflow
    
- n8n
    
- Zapier
    
- CI/CD 流程
    
- 企业中间件
    
- 其他微服务  
    只要能发 HTTP 请求，就能调这个 API。  
    所以它很像把模型能力“产品化”了：谁都可以通过统一接口接入。Databricks 官方对 serving endpoint 的标准调用路径就是 REST invocations 接口。([Databricks Documentation](https://docs.databricks.com/aws/en/machine-learning/model-serving/score-custom-model-endpoints?utm_source=chatgpt.com "Query serving endpoints for custom models"))

## 你可以把它这样理解

### 没有 API 的时候

你得在某个 notebook 里：

- 手动加载模型
    
- 手动准备输入
    
- 手动执行预测
    
- 手动拿结果
    

这更像“人用”。

### 有了 API 之后

任何系统都可以这样理解：

> 给这个地址发请求  
> 把输入传进去  
> 它把结果返回出来

这更像“系统用”。

## 一个最实际的例子

假设你做了一个文本分类 endpoint：

```text
sentiment-prod-endpoint
```

那它可以这样被不同地方用：

- **网站后端**：用户提交评论，后端调用 endpoint 判断正负面
    
- **客服系统**：自动分析工单情绪
    
- **Databricks SQL**：批量给历史评论打标签
    
- **Notebook**：测试模型新版本
    
- **外部 Python 服务**：每天批量跑数据
    

也就是说，**endpoint 不是给人手工点着玩的，主要是给程序、系统、工作流调用的。**
