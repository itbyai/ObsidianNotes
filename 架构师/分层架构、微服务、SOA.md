# 软件架构基础

## 1. 分层架构 (Layered Architecture)
### 概念
- 将系统按照职责划分为多个层，每层只与相邻层交互
- 常见层次：
  1. **表示层（Presentation Layer）**：用户界面和交互
  2. **业务逻辑层（Business Layer / Domain Layer）**：核心业务规则
  3. **数据访问层（Data Access Layer / Persistence Layer）**：数据库或存储操作
  4. **基础设施层（Infrastructure Layer）**：网络、文件系统、外部服务

### 特点
- 明确分工，职责清晰
- 易于理解和维护
- 层次间的依赖严格控制（通常上层依赖下层）

### 优点
- 易于开发和测试
- 高内聚低耦合
- 模块化管理

### 缺点
- 层与层之间通信可能带来性能开销
- 随系统复杂度增长，层级过多可能导致修改不灵活
- 不适合非常高并发或复杂分布式系统

---

## 2. 微服务架构 (Microservices Architecture)
### 概念
- 将系统拆分为 **小型、自治、独立部署的服务**
- 每个微服务专注一个业务功能，通过 **HTTP/REST、gRPC、消息队列** 等进行通信
- 服务自治：自己的数据库和数据模型

### 特点
- 每个服务独立开发、部署、扩展
- 服务可以使用不同语言或技术栈
- 强调 **去中心化** 和 **服务自治**

### 优点
- 高度可扩展，可针对热点服务单独扩容
- 提高团队开发效率（多团队并行开发）
- 弹性部署，支持 CI/CD

### 缺点
- 系统复杂度高（分布式问题：网络延迟、事务一致性）
- 调试和监控难度增加
- 需要成熟的 DevOps 和自动化部署能力

### 适用场景
- 大型复杂系统
- 业务模块独立且频繁迭代
- 需要高可扩展性和弹性部署

---

## 3. 面向服务架构 (SOA, Service-Oriented Architecture)
### 概念
- 将系统划分为 **功能服务（Service）**，通过 **企业服务总线（ESB）或消息总线** 进行通信
- 服务重用为核心目标
- 强调 **契约、接口和服务组合**

### 特点
- 服务通常粗粒度，面向企业业务
- 注重跨系统、跨组织的整合
- 常用 SOAP、WSDL、ESB 等技术

### 优点
- 支持系统间的互操作和整合
- 服务重用率高
- 适合企业级应用（ERP、CRM、供应链系统）

### 缺点
- 初期架构成本高
- 重耦合的 ESB 可能成为性能瓶颈
- 不适合快速迭代或小型系统

### 适用场景
- 大型企业级系统
- 系统之间需要高度集成和服务重用
- 长生命周期、业务稳定的系统

---

## 架构对比总结

| 特性 | 分层架构 | 微服务 | SOA |
|------|----------|--------|-----|
| 粒度 | 中等（按功能划层） | 细粒度（按业务功能拆服务） | 粗粒度（按企业业务服务） |
| 服务独立性 | 低 | 高 | 中等 |
| 部署方式 | 整体部署 | 独立部署 | 可组合部署 |
| 技术栈限制 | 一致性较高 | 可多样化 | 可多样化，但需契约规范 |
| 适用场景 | 中小型应用，简单系统 | 大型复杂业务系统 | 企业系统整合，跨系统集成 |
