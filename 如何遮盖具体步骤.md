降低 PII 风险，最常见的做法是：

**在数据进入向量库、日志、模型之前，先识别，再遮盖，再按需要保留最少信息。**

---

# 1. 常见遮盖方式

## 全量替换

把敏感字段直接替换成占位符：

- 邮箱 → `[EMAIL]`
    
- 电话 → `[PHONE]`
    
- 身份证 → `[ID]`
    

例子：

```text
原文：请联系 zhangsan@example.com 或 0412 345 678
结果：请联系 [EMAIL] 或 [PHONE]
```

这种最安全，最适合：

- RAG 语料入库前
    
- 发给大模型前
    
- 普通日志
    

---

## 部分遮盖

保留少量可识别信息，其余打码：

- 邮箱：`z***@example.com`
    
- 电话：`0412***678`
    
- 身份证：`1101********1234`
    

例子：

```text
原文：zhangsan@example.com
结果：z***@example.com
```

这种适合：

- 客服界面
    
- 审计界面
    
- 需要人工核对但不想暴露全量信息
    

---

## 令牌化

把真实值替换成 token，再单独保存映射关系：

- `zhangsan@example.com` → `EMAIL_001`
    
- `0412345678` → `PHONE_001`
    

例子：

```text
原文：请联系 zhangsan@example.com
结果：请联系 [EMAIL_001]
```

映射表另存：

|token|real_value|
|---|---|
|EMAIL_001|[zhangsan@example.com](mailto:zhangsan@example.com)|

这种适合：

- 企业系统
    
- 后续还原有业务需要
    
- 严格权限控制场景
    

---

# 2. 一般处理流程

## 第一步：识别 PII

常见识别对象：

- 电话号码
    
- 邮箱地址
    
- 身份证号
    
- 银行卡号
    
- 姓名
    
- 地址
    
- 医疗号
    
- 护照号
    

识别方式通常有两类：

### 规则识别

用正则表达式、字典、关键词。

适合：

- 电话
    
- 邮箱
    
- 身份证
    
- 银行卡
    

### 模型识别

用 NER / PII detection 模型识别人名、地址、机构名等。

适合：

- 复杂文本
    
- 非固定格式字段
    
- 多语言场景
    

---

## 第二步：决定遮盖策略

不同字段可以不同处理：

- 电话、邮箱：全量替换或部分遮盖
    
- 姓名：可替换为 `[NAME]`
    
- 地址：可只保留城市
    
- 银行卡：只留后四位
    
- 医疗号：全量替换
    

---

## 第三步：在进入模型前处理

最推荐放在这里：

```text
原始文本 -> PII识别 -> 遮盖/令牌化 -> chunking -> embedding -> 向量库 / LLM
```

这样最稳，因为：

- 敏感信息不会进向量库
    
- prompt 里不会带原始 PII
    
- 日志里也更干净
    

---

# 3. Python 直接实现

## 全量替换版

```python
import re

EMAIL_PATTERN = r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b'
PHONE_PATTERN = r'(?:(?:\+?\d{1,3}[-.\s]?)?(?:\(?\d{2,4}\)?[-.\s]?)?\d{3,4}[-.\s]?\d{3,4})'

def mask_pii(text: str) -> str:
    text = re.sub(EMAIL_PATTERN, "[EMAIL]", text)
    text = re.sub(PHONE_PATTERN, "[PHONE]", text)
    return text

sample = "请联系 zhangsan@example.com 或 0412 345 678"
print(mask_pii(sample))
```

输出：

```text
请联系 [EMAIL] 或 [PHONE]
```

---

## 部分遮盖版

```python
import re

def partial_mask_email(text: str) -> str:
    pattern = r'\b([A-Za-z0-9._%+-])([A-Za-z0-9._%+-]*)(@[A-Za-z0-9.-]+\.[A-Za-z]{2,})\b'
    return re.sub(pattern, r'\1***\3', text)

def partial_mask_phone(text: str) -> str:
    def repl(match):
        raw = match.group(0)
        digits = re.sub(r'\D', '', raw)
        if len(digits) < 7:
            return "[PHONE]"
        return digits[:4] + "***" + digits[-3:]
    pattern = r'(?:(?:\+?\d{1,3}[-.\s]?)?(?:\(?\d{2,4}\)?[-.\s]?)?\d{3,4}[-.\s]?\d{3,4})'
    return re.sub(pattern, repl, text)

sample = "请联系 zhangsan@example.com 或 0412 345 678"
sample = partial_mask_email(sample)
sample = partial_mask_phone(sample)
print(sample)
```

输出可能是：

```text
请联系 z***@example.com 或 0412***678
```

# 4. Databricks / PySpark 里怎么做

如果你在表数据里处理，通常这样写：

```python
from pyspark.sql import functions as F

email_pattern = r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b'
phone_pattern = r'(?:(?:\+?\d{1,3}[-.\s]?)?(?:\(?\d{2,4}\)?[-.\s]?)?\d{3,4}[-.\s]?\d{3,4})'

masked_df = (
    df
    .withColumn("masked_text", F.regexp_replace("text", email_pattern, "[EMAIL]"))
    .withColumn("masked_text", F.regexp_replace("masked_text", phone_pattern, "[PHONE]"))
)
```

这适合：

- 批量清洗 RAG 文档
    
- 日志脱敏
    
- 入湖前预处理

# 5. 在 RAG 里最推荐的放置点

## 最佳位置

在 **embedding 和索引之前** 做。

```text
原始文档
-> 清洗
-> PII 遮盖
-> chunking
-> embedding
-> vector search
-> LLM
```

## 为什么

如果先入库再遮盖，会有几个风险：

- 向量库里已经存了原始 PII
    
- 检索时会把 PII 召回出来
    
- 模型可能复述敏感信息
    

---

# 6. 工程上更稳的做法

## 输入前遮盖

保护向量库和模型输入。

## 输出后再扫一次

防止模型根据上下文“补全”或复述出敏感信息。

## 日志也要脱敏

很多系统最大风险不是模型，而是：

- request log
    
- response log
    
- debug log
    
- trace
    

---

# 7. 常见坑

## 正则太宽

会把订单号、编号误判成电话。

## 正则太窄

国际号码、带括号号码漏识别。

## 只遮盖输入，不遮盖输出

模型可能还是把敏感信息说出来。

## 只靠 prompt

比如写“不要输出电话”，这不够稳。  
真正有效的是静态脱敏。

# 8. 一句最实用的总结

**遮盖敏感字段的核心，就是先识别 PII，再按字段类型做全量替换、部分遮盖或令牌化，并且尽量在进入向量库、日志和模型之前完成。**

不一定，**看你用的是哪一种处理方式**。

## 1. 直接遮盖

例如：

- `zhangsan@example.com` → `[EMAIL]`
    
- `0412345678` → `[PHONE]`
    

这种如果你**只保存了遮盖后的结果**，那基本就**恢复不了**了。  
因为原值已经被你覆盖掉了。

也就是说：

- 遮盖后文本还在
    
- 原始值没单独保存
    
- 那就是**永久丢失**
    

---

## 2. 部分遮盖

例如：

- `zhangsan@example.com` → `z***@example.com`
    
- `0412345678` → `0412***678`
    

这种也通常**不能完整恢复**。  
因为中间那部分字符已经没了。

你只能看出一部分特征，但不能准确还原原值。

所以它本质上也是：

**可识别一点点，但不可完整恢复。**

---

## 3. 令牌化

例如：

- `zhangsan@example.com` → `[EMAIL_001]`
    
- `0412345678` → `[PHONE_001]`
    

然后你另外保存一张映射表：

|token|real_value|
|---|---|
|EMAIL_001|[zhangsan@example.com](mailto:zhangsan@example.com)|
|PHONE_001|0412345678|

这种是**可以恢复**的。  
因为你不是把原值“删掉”，而是把原值“移到受控位置”。

恢复条件是：

- 映射表还在
    
- 你有权限访问映射表
    

所以这种不叫真正“丢失”，而是：

**脱敏展示 + 安全可逆。**

---

## 4. 加密

如果你把敏感字段加密保存，比如：

- 原文存密文
    
- 展示时只给脱敏结果
    

那也是**可以恢复**的，前提是你有密钥。

这种和令牌化有点像，但思路不同：

- **令牌化**：靠映射表恢复
    
- **加密**：靠密钥解密恢复
    

---

# 一句话总结

## 会不会永远丢失，取决于方法：

- **全量遮盖**：通常不可恢复，原值等于丢了
    
- **部分遮盖**：通常也不可完整恢复
    
- **令牌化**：可恢复
    
- **加密**：可恢复
    

---

# 在 RAG / LLM 场景里通常怎么选

## 如果目标是尽量降低 PII 风险

一般会选：

- 全量遮盖
    
- 或部分遮盖
    

因为这样更安全，哪怕向量库泄露，也看不到原始值。

## 如果业务上后面还要找回真实值

一般会选：

- 令牌化
    
- 或加密 + 单独权限控制
    

---

# 最实用的理解

你可以这样记：

**“遮盖”通常是为了不让模型和向量库看到原文；如果还想恢复，就不要只做纯遮盖，而要做令牌化或加密。**

# 一个最短例子

## 不可恢复

```text
张三电话是 0412345678
-> 张三电话是 [PHONE]
```

如果没留原表，就没了。

## 可恢复

```text
张三电话是 0412345678
-> 张三电话是 [PHONE_001]
```

并且另存：

```text
PHONE_001 -> 0412345678
```

这样以后能恢复。
