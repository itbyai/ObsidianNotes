# Databricks Generative AI Engineer 讲义（完整修正版）

## 总体说明

这份讲义围绕生成式 AI 应用的完整生命周期展开，内容覆盖七个核心模块：

- Design Applications
    
- Data Preparation
    
- Application Development
    
- Assembling and Deploying Applications
    
- Governance
    
- Evaluation and Monitoring
    

整体主线非常清晰：

**先把业务需求翻译成正确的应用设计，再把数据准备好，再把应用开发出来，然后组装与部署上线，同时做好治理、评估与持续监控。**

这不是单纯的模型知识汇总，而是一套从**需求设计 → 数据准备 → 应用开发 → 系统部署 → 治理控制 → 评估监控**的完整工程化方法论。

---

# 一、整体框架

生成式 AI 应用从落地角度看，可以拆成七个核心问题。

## 1. 业务需求是否被正确翻译成应用设计

重点包括：

- 应该让模型完成什么任务
    
- 应该用什么 prompt
    
- 应该选什么链式结构
    
- 是否需要检索
    
- 是否需要多工具协作
    

## 2. 数据是否准备好了

重点包括：

- 文档是否干净
    
- 是否选对知识源
    
- 是否做好切分
    
- 是否便于 embedding、检索和训练
    

## 3. 应用是否设计合理

重点包括：

- prompt 是否清晰
    
- 链式组件是否选对
    
- agent 是否能调用工具
    
- 模型是否符合成本与延迟约束
    

## 4. 系统是否能真正部署起来

重点包括：

- 模型如何封装
    
- 检索如何接入
    
- 向量索引如何构建
    
- 服务如何暴露给外部调用
    

## 5. 是否满足治理要求

重点包括：

- 是否泄露 PII
    
- 是否容易被 prompt injection 利用
    
- 是否使用了高风险数据源
    
- 是否对有害内容做了治理
    

## 6. 上线前是否评估充分

重点包括：

- 是否选对评估指标
    
- 是否能比较不同模型与方案
    
- 是否记录可复盘的实验数据
    

## 7. 上线后是否持续可观察

重点包括：

- 线上表现有没有退化
    
- 成本是否可控
    
- 响应时间和用户体验是否稳定
    
- 检索质量是否下降
    

---

# 二、Design Applications

## 模块定位

Design Applications 关注的是：

**如何把业务需求翻译成合适的生成式 AI 应用设计。**

重点不是底层部署，而是站在设计层判断：

- 应该用什么 prompt
    
- 应该让模型完成什么任务
    
- 应该选什么链式组件
    
- 输入和输出如何定义
    
- 多步骤 agent 如何安排工具顺序
    

---

## 2.1 通过 prompt 明确约束输出格式

当应用要求模型输出固定格式结果时，更好的做法不是只说“总结一下”，也不是笼统要求“返回 JSON”，而是：

- 直接写明目标格式
    
- 明确字段
    
- 明确结构
    
- 最好给出精确模板
    

这种方式的价值在于：

- 提高输出一致性
    
- 提高程序可解析性
    
- 降低歧义
    
- 更容易约束 hallucination
    

### 核心原则

**输出越需要标准化，prompt 就越要具体、结构化。**

---

## 2.2 根据业务目标选择正确的模型任务类型

应用设计首先要做的是把业务需求映射成合适的任务类型。

例如，当需求是分析产品评论、识别新问题和负面趋势时，更贴切的任务通常是：

- Sentiment Analysis
    

而不一定是：

- 普通摘要
    
- 命名实体识别
    
- 其他不直接对应的任务
    

### 核心原则

**先确定业务任务类型，再决定模型该做什么。**

---

## 2.3 基于文档回答问题时，应优先采用 RAG 型结构

如果应用要求：

- 基于内部文档回答
    
- 回答必须 grounded
    
- 答案要依赖政策、知识库或已有文档
    

那么更合理的设计通常是 RAG 流程：

- Retriever
    
- Prompt
    
- LLM
    

也就是：

- 先检索
    
- 再组织上下文
    
- 最后生成答案
    

### 核心原则

**题目一旦强调“基于文档回答”，就优先想到 RAG，而不是让 LLM 裸答。**

---

## 2.4 将业务目标准确翻译成输入与输出规格

应用设计中很常见的问题不是模型能力不足，而是：

- 输入定义错了
    
- 输出规格偏了
    
- 没有忠于题干
    

例如，如果任务是读取 earnings transcripts 并输出风险与机会，那么：

### 输入

- earnings transcripts
    

### 输出

- risks
    
- opportunities
    
- 最好是结构化结果
    

而不是把输入偷换成公司名、股票代码，或者把输出偷换成单一分数。

### 核心原则

**应用设计要忠于真实业务需求，输入输出定义必须准确。**

---

## 2.5 多步骤 agent 的工具顺序要符合真实工作流

如果一个 agent 需要：

- 搜索公司财务信息
    
- 总结结果
    
- 发送邮件
    

那么合理顺序应是：

1. 搜索
    
2. 处理
    
3. 执行
    

例如：

1. Web Search
    
2. Summarizer
    
3. Email Tool
    

### 核心原则

**多工具代理应遵循“取数 → 处理 → 输出/执行”的自然链路。**

---

## 2.6 本模块方法论

Design Applications 的主线很统一：

- 应用设计首先是需求翻译
    
- prompt 不仅影响内容，也影响结构
    
- 文档型问答优先考虑检索增强
    
- 多步骤应用本质上是工作流设计
    

### 速记

- 格式严格，prompt 更严格
    
- 任务先分类，再选模型动作
    
- 文档问答优先 RAG
    
- 工具顺序按真实流程排
    

---

# 三、Data Preparation

## 模块定位

Data Preparation 的目标不是简单把文档导入系统，而是让数据以最适合检索、生成、训练和评估的方式进入系统。

这一部分的核心能力包括：

- 根据文档结构选择合适的 chunking strategy
    
- 清理无关内容，减少噪音
    
- 针对不同文档类型选对提取工具
    
- 将处理结果结构化写入 Delta Lake / Unity Catalog
    
- 为 RAG 选择高质量知识源
    
- 准备与任务严格对齐的训练样本
    
- 用合适指标评估 retrieval quality
    
- 理解 advanced chunking 和 re-ranking
    

---

## 3.1 根据文档结构设计 chunking strategy

### 结构化长文档应优先按逻辑结构分块

如果文档本身已经有明确标题或章节，例如法律文档、制度文档、政策文档，那么更合理的切分方式不是固定长度切块，而是：

- 按 section
    
- 按 heading
    
- 按逻辑边界
    

这样做的原因是：

- 保留语义完整性
    
- 避免切断关键上下文
    
- 提高检索相关性
    
- 更适合长篇结构化文档
    

### FAQ / Q&A 文档应按问答对切块

如果文档本身就是问答结构，那么最自然的语义单元就是：

- 一个 question
    
- 一个对应 answer
    

这种切分方式更贴近用户真实的检索方式，也更容易命中目标答案。

### 技术文档应按语义块切分

对于技术手册、系统说明、开发文档，文档中往往包含：

- 代码块
    
- 图表
    
- 参数说明
    
- 配置说明
    
- 解释性文字
    

更适合按语义块切，而不是机械按 token 数切。

### 长文档不要整篇只做一个 embedding

当文档很长，比如 10K tokens，而查询很短时，应将文档切成较小的 spans 后再 embedding。  
分段嵌入更有利于短查询命中具体相关片段。

### chunking 方案要靠检索指标验证

不同 chunking 方案不能只靠直觉判断。  
如果 section-based chunks 的 Recall@K 更高，就说明它在该场景中更适合。

---

## 3.2 预处理无关内容，提升 embedding 和 retrieval 质量

很多文档中会出现大量重复且无知识价值的内容，例如：

- Confidential
    
- Page x of y
    
- 页眉页脚
    
- 固定声明
    
- 重复模板文本
    

这些内容会污染 embedding，降低 retrieval relevance，因此在预处理阶段应优先清理。

### 核心原则

**不是所有文字都值得进入检索系统。**

---

## 3.3 针对文档类型选择合适的提取工具

### 扫描版 PDF 需要 OCR

如果 PDF 是扫描型、图像型，而不是原生文本 PDF，那么应使用 OCR 工具提取文本，例如：

- Tesseract OCR
    

### 文本型 PDF 与扫描型 PDF 要区分

- 文本型 PDF：可以直接解析
    
- 扫描型 PDF：需要 OCR
    

---

## 3.4 将 chunked text 结构化写入 Delta Lake

文档经过提取与分块后，不应以松散文本形式直接落地，而应先转换为结构化数据，再写入表。

### 推荐流程

1. 将 chunked text 组织成记录
    
2. 加入元数据，例如：
    
    - chunk_id
        
    - text
        
    - source
        
    - page
        
    - section
        
    - metadata
        
3. 转换成 DataFrame
    
4. 写入 Unity Catalog 管理下的 Delta Lake 表
    

---

## 3.5 为 RAG 选择高质量知识源

RAG 的质量高度依赖知识源本身。

### 高风险领域优先选择权威资料

如果是医疗、法律、金融、政策等高风险领域，应优先选择：

- 官方指南
    
- 经验证政策
    
- 规范文档
    
- 权威内部知识文档
    

而不应优先依赖：

- 社交媒体评论
    
- 营销材料
    
- 未经验证讨论
    
- 来源不明内容
    

### 法律风险最低的数据源

企业内部文档通常比竞争对手资料、抓取内容或未授权第三方资料风险更低。

---

## 3.6 训练数据中的 prompt-response pair 必须与任务严格对齐

如果目标任务是 summarization，那么训练样本必须体现：

- prompt 明确要求 summary
    
- response 真实输出 summary
    

不合适的训练样本包括：

- prompt 和 response 任务不匹配
    
- 两者完全无关
    
- 形式相关但目标偏离，例如写诗而不是总结
    

### 核心原则

**输入和输出都必须服务于同一个目标任务。**

---

## 3.7 评估 retrieval quality 的指标

用于评估 retriever 能否把相关文档找出来的典型指标包括：

- Precision@K
    
- Recall@K
    

其中 Precision@K 关注的是：

**前 K 个结果中有多少是真正相关的。**

Precision / Recall 这类指标都依赖 ground truth。

---

## 3.8 re-ranking 的作用

re-ranking 的目的不是生成答案，也不是重新切块，而是：

- 对初始检索结果进行重新排序
    
- 根据更深层语义相关性做精排
    
- 提高送入模型的上下文质量
    

可以理解为：

- 初始 retrieval = 粗筛
    
- re-ranking = 精排
    

---

# 四、Application Development

## 模块定位

Application Development 关注的是如何把模型能力变成一个真正可用的应用。  
重点不仅是“能回答”，还包括：

- 能检索
    
- 能调用工具
    
- 能遵守约束
    
- 能控制风险
    
- 能满足成本与延迟要求
    

---

## 4.1 为特定数据需求选择合适的处理工具

如果需求是从 S3 中读取大量 CSV、处理每日销售数据并送入 RAG workflow，那么更合适的方案是：

- 使用 PySpark 作业读取 S3
    
- 处理结构化数据
    
- 输出下游友好的结果
    

这比简单脚本更适合数据工程型任务。

---

## 4.2 选择适合 RAG 应用的链式组件

当应用既需要检索，又需要给出有依据的回答时，更合适的组件是：

- Retrieval QA with Sources Chain
    

它的优势在于：

- 自带 retrieval
    
- 能结合 LLM
    
- 输出可追溯来源
    
- 更适合 grounded QA
    

---

## 4.3 Prompt 设计决定输出形式和结果质量

更好的 prompt 不仅要说明任务，还要明确规定输出形式。  
例如要求模型总结政策文件时，应明确：

- 做 summary
    
- 用 bullet points
    
- 给出 TLDR
    
- 限定条数或结构
    

### 核心原则

**不要只告诉模型做什么，还要告诉它怎么输出。**

---

## 4.4 自然语言请求应增强为结构化上下文

例如用户说：

- Show me Q3 expenses for project Phoenix
    

更好的做法是先提取并结构化出：

- quarter = Q3
    
- project = Phoenix
    
- metric = expenses
    

这样更利于：

- 工具调用
    
- 数据定位
    
- 减少歧义
    
- 生成更准确回答
    

---

## 4.5 多步 prompt 设计要有明确约束

如果第一步先生成 200-word summary，第二步要压缩，则更有效的第二步提示应写成：

- Shorten the summary to no more than 50 words.
    

重点是给出明确、可执行的硬性约束。

---

## 4.6 敏感领域应通过显式 guardrail 控制风险

如果不希望应用提供法律建议，更合适的做法是：

- 在 prompt logic 中加入条件拒绝规则
    
- 遇到法律建议类请求时直接拒绝
    

这通常比先做微调更直接、可控、可审计。

---

## 4.7 用 meta prompt 降低 hallucination

更安全的 meta prompt 往往会要求模型：

- 仅根据提供的上下文回答
    
- 如果不知道，就明确说 I don’t know
    

这样可以减少：

- 自行脑补
    
- 凭记忆作答
    
- 不确定情况下的虚构内容
    

---

## 4.8 agent 的函数调用需要显式声明接口

如果要让 agent 调用函数，提示中应明确告诉它：

- 可用函数有哪些
    
- 每个函数的签名是什么
    
- 在什么情况下应该调用
    

函数调用能力不是靠暗示，而是靠结构化接口说明。

---

## 4.9 多步骤代理的关键能力是 tool routing

一个 multi-step agent 可能需要：

- 查询数据库
    
- 执行 web search
    
- 发送 email
    

此时关键能力之一就是：

- Tool Routing
    

也就是系统能够决定下一步该调用哪个工具、按什么顺序调用。

---

## 4.10 模型选择要服务于业务约束

### 低成本、低延迟的实时场景

如果要求：

- real-time answers
    
- constrained latency
    
- low cost
    

那么更轻量、速度更快、成本更低的模型更实用。

### 选模型不能只看流行度

更合理的模型选择维度包括：

- 是否适合任务
    
- token limit 是否满足需求
    
- 延迟是否可接受
    
- 成本是否在预算内
    
- 是否有可靠评估结果
    
- 是否有与任务相关的 metadata 和 performance metrics
    

---

## 4.11 隐私泄露是应用设计中的重要风险点

如果聊天机器人回答里包含某个用户的个人信息，即使信息已经过时，这也应优先归类为：

- privacy leak
    

说明应用开发不仅是能力设计，也是风险边界设计。

---

# 五、Assembling and Deploying Applications

## 模块定位

这一部分关注如何把模型、链路、向量检索、封装逻辑和服务治理真正组装成一个可部署系统。

核心问题包括：

- 怎么封装模型
    
- 怎么注册模型
    
- 怎么建索引
    
- 怎么部署服务
    
- 怎么做批量推理
    

---

## 5.1 用 MLflow PyFunc 封装模型

在 MLflow PyFunc wrapper 中，最核心的两个方法通常是：

- load_context
    
- predict
    

其中：

- `load_context` 用于加载依赖和工件
    
- `predict` 用于执行推理
    

预处理和后处理通常写在 `predict` 内部。

---

## 5.2 Databricks 模型服务权限治理依赖 Unity Catalog

Databricks 中 model serving endpoint 的访问控制，应放在 Unity Catalog 权限治理框架中理解。

Unity Catalog 管理的不只是表，还包括：

- 模型
    
- 端点
    
- 数据访问边界
    

---

## 5.3 LangChain 中简单顺序链的本质是链的组合

如果有一个简单函数，例如把输入转成大写，那么要接入 `SimpleSequentialChain`，本质上需要：

- 先把函数封装成 chain
    
- 再把 chain 串成顺序链
    

顺序链的基本单位是 chain，而不是普通裸函数。

---

## 5.4 Prompt 与模型组合的基础组件是 LLM Chain

当需求是“prompt + LLM 输出”的标准组合时，最基础组件之一就是：

- LLM Chain
    

---

## 5.5 基础 RAG 应用的必需组件是 embedding model

一个标准 RAG 应用至少需要：

- 文档向量化能力
    
- 查询向量化能力
    
- 相似度检索能力
    

因此 embedding model 是基础必需组件。

---

## 5.6 用 MLflow 注册模型到 Unity Catalog

模型进入 Unity Catalog，不是简单保存成文件，而是需要使用正式的 MLflow 模型注册流程：

- 指定模型 URI
    
- 指定注册名称
    
- 进入统一的 registry 管理
    

---

## 5.7 RAG 应用的典型部署顺序

一个标准 RAG app endpoint 的部署顺序可以概括为：

1. 处理文档
    
2. 生成 embeddings
    
3. 建立索引
    
4. 构建 retriever
    
5. 注册模型或应用
    
6. 创建 serving endpoint
    

核心逻辑是：  
先把知识层和检索层准备好，再对外暴露服务。

---

## 5.8 Mosaic AI Vector Search 的标准查询方式是 similarity search

Vector Search 的核心工作方式是：

- 输入 query vector
    
- 在索引中做相似度搜索
    
- 返回 top-k 结果
    

---

## 5.9 外部大模型 API 场景下，部署的是包装层而不是基础模型本体

如果应用调用的是外部 LLM API，那么在 Databricks 中部署的不是模型本体，而是：

- wrapper chain
    
- agent
    
- orchestration layer
    

也就是负责组织输入、调用外部 API、整合检索和返回结果的应用逻辑层。

---

## 5.10 支撑 RAG 服务的关键资源

在 Databricks 上服务 RAG 应用时，关键资源包括：

- Vector Search
    
- MLflow Model Registry
    
- Model Serving
    

三者分别支撑：

- 检索
    
- 注册与版本管理
    
- 对外服务
    

---

## 5.11 Mosaic AI Vector Search 的边界

Mosaic AI Vector Search 可以负责：

- dense vector storage
    
- embedding indexing
    
- fast similarity search
    

但不负责：

- 动态现场生成 embeddings
    

也就是它解决的是“存、索、搜”，不是“生成 embedding”。

---

## 5.12 embeddings 存到 Delta 后，还要建立向量索引

把 embeddings 写入 Delta 表只是第一步。  
真正支持高效语义检索的关键是：

- 在 embedding 列上建立 vector search index
    

---

## 5.13 对 Delta 表上的大规模记录做批量推理

如果要对大量记录运行摘要模型，更高效的方法是：

- 使用 AI Query
    
- 直接在 SQL / 表层面做 batch inference
    

这通常比逐条 notebook 调用更适合批量场景。

---

# 六、Governance

## 模块定位

Governance 关注的是生成式 AI 应用在真实场景中的安全、合规、隐私、法律和内容边界。

重点不是让模型更强，而是让系统在：

- 使用数据
    
- 接收输入
    
- 输出结果
    

时始终处于可控边界内。

---

## 6.1 数据准备阶段应优先遮盖敏感字段

在客户支持类应用中，为降低 PII 风险，应在数据准备阶段有针对性地遮盖敏感字段，例如：

- 电话号码
    
- 邮箱地址
    
- 其他可识别个人身份的信息
    

这比删除全部用户数据更合理，因为既保留业务价值，又降低暴露风险。

---

## 6.2 防 prompt injection 的核心是输入治理

面向公众的 GenAI 应用需要防范恶意输入。  
最直接有效的防护思路是：

- 清洗用户输入
    
- 验证用户输入
    
- 过滤有害指令
    

而不是单纯依赖：

- 限制上下文窗口
    
- 换小模型
    
- 一刀切屏蔽所有特殊字符
    

---

## 6.3 对数据集中的 offensive content 应在预处理阶段过滤

如果 RAG 数据集中包含冒犯性、有害或不适宜内容，更合理的做法是：

- 在预处理阶段用文本过滤器排除它们
    

而不是试图通过增大 chunk size、减少 top-k 或忽略问题来掩盖。

---

## 6.4 选择法律风险更低的数据源

企业内部文档通常比以下来源法律风险更低：

- 竞争对手专有内容
    
- 未授权抓取内容
    
- 未经许可复制的付费内容
    

### 核心原则

**知识源不仅要有用，还必须合法、授权清晰。**

---

## 6.5 大规模 toxic language 治理要依赖自动化分类

如果训练数据中已知存在 toxic language，而数据量又很大，那么更可扩展的方法是：

- 使用文本分类模型自动检测
    
- 自动剔除有问题内容
    

这比完全依赖人工审核更可持续。

---

## 6.6 对无关但会干扰推理的字段使用占位符遮盖

如果数据里包含大量无关字段，例如：

- 日期
    
- 订单号
    
- 无意义结构化 ID
    

更合理的处理方式通常是：

- 用占位符替换，例如 `ORDER_ID`
    

这样既保留结构，又减少噪音。

---

# 七、Evaluation and Monitoring

## 模块定位

Evaluation and Monitoring 关注的是：

- 上线前如何比较方案、判断是否够好
    
- 上线后如何持续观察系统是否稳定、准确、可控、成本合理
    

这是生成式 AI 生命周期中非常关键的质量闭环。

---

## 7.1 模型选择应由任务最关键的指标主导

例如法律文档摘要任务，最关键的不是模型受欢迎程度，也不是最便宜，而是：

- factual consistency
    
- fact-level accuracy
    
- 在可接受延迟下的真实性表现
    

如果任务核心要求是 factual correctness，那么 factual consistency 应优先于热度、体积或纯成本。

---

## 7.2 生产级 customer-facing 应用要持续监控关键运营指标

更重要的线上指标包括：

- response time
    
- user engagement
    
- success-related user behavior
    

相比之下，embeddings 数量、训练数据大小等，不是这里最核心的用户侧运营监控指标。

---

## 7.3 用 MLflow 记录结构化评估信息

在 RAG 或 LLM 应用评估中，MLflow 适合记录：

- model latency
    
- accuracy / quality metrics
    
- retrieval outputs
    
- 示例结果
    
- 不同版本比较信息
    

它的价值在于帮助团队：

- 复盘
    
- 比较
    
- 跟踪变化
    
- 观察不同配置表现
    

---

## 7.4 inference logging 是生产可观察性的基础

Inference logging 的关键价值是持续记录：

- queries
    
- responses
    
- retrieval quality over time
    
- 真实线上行为
    

如果没有这些日志，就很难知道：

- 质量是否在退化
    
- 某类问题是否经常失败
    
- 用户查询分布是否变化
    
- 检索是否越来越不准
    

---

## 7.5 推理成本控制依赖 serving concurrency 和扩缩容管理

在 RAG 场景中，控制成本的重要抓手之一是：

- model serving concurrency limits
    
- scaling control
    
- resource usage management
    

核心逻辑是：  
服务层的并发与扩缩容设置，直接决定资源消耗和成本走势。

---

## 7.6 inference tables 和 agent monitoring 的主要作用

它们的重点是长期跟踪：

- LLM performance
    
- query success
    
- error trends
    
- agent failure patterns
    
- 随时间变化的质量问题
    

它们不是主要用来做 embeddings 存储或模型注册，而是为线上应用提供运行质量可观察性。

---

## 7.7 哪些评估方法依赖 ground truth

需要 ground truth 的典型指标包括：

- precision
    
- recall
    

因为必须先知道什么是正确答案、什么是相关项。  
相对地，用户反馈、人工偏好排序或 LLM-as-a-judge 不一定严格依赖 ground truth。

---

## 7.8 Evaluation 与 Monitoring 的区别

最核心的区别可以概括为：

- Evaluation：通常在部署前进行
    
- Monitoring：部署后持续进行
    

可以简单理解为：

- Evaluation = 上线前验收
    
- Monitoring = 上线后体检
    

---

# 八、统一方法论

把整份讲义串起来，可以总结出一套非常稳定的方法论。

## 8.1 先把需求设计正确，再谈系统实现

很多生成式 AI 问题，本质不是模型不够强，而是：

- 任务定义错了
    
- 输出格式不清楚
    
- 不该用裸答却没做检索
    
- 多工具流程设计混乱
    

---

## 8.2 先把数据准备好，再谈模型效果

很多问题也不是模型本身造成的，而是：

- 文档没选好
    
- 噪音没清掉
    
- chunking 不合理
    
- 训练样本不对齐
    

---

## 8.3 结构化设计优于模糊设计

整套内容反复强调结构化思维：

- chunk 要按结构切
    
- prompt 要明确格式
    
- 自然语言请求要结构化增强
    
- agent 要显式声明函数
    
- 多工具任务要做 tool routing
    

---

## 8.4 应用落地不仅是“会答”，更是“可控”

一个真正可用的应用必须同时具备：

- 能力
    
- 安全
    
- 治理
    
- 部署
    
- 监控
    
- 成本管理
    

---

## 8.5 质量必须可评估、可比较、可观察

这包括：

- 用正确指标做评估
    
- 用 MLflow 做结构化记录
    
- 用 inference logging 做线上观察
    
- 用 monitoring 追踪长期变化
    

---

## 8.6 生产系统必须考虑取舍

很多设计都不是单纯追求“最强”，而是要平衡：

- 质量 vs 成本
    
- 延迟 vs 能力
    
- 安全 vs 灵活性
    
- 自动化 vs 人工审查
    
- 召回范围 vs 精度
    

---

# 九、总复习清单

## Design Applications

- 输出格式越严格，prompt 越要明确写出结构
    
- 业务目标应先映射为正确任务类型
    
- 基于文档回答问题，优先采用 RAG 结构
    
- pipeline 设计时要准确区分输入与输出
    
- 多步骤 agent 应按“搜索 → 处理 → 执行”的顺序安排工具
    

## Data Preparation

- 长文档按结构切
    
- FAQ 按问答对切
    
- 技术文档按语义块切
    
- 扫描 PDF 用 OCR
    
- 页眉页脚等噪音应预清洗
    
- chunk 先结构化再落 Delta
    
- RAG 优先选择权威知识源
    
- prompt-response pair 必须与任务对齐
    
- retrieval 常用 Precision@K / Recall@K
    
- re-ranking 用于精排
    

## Application Development

- S3 批量 CSV 处理优先 PySpark
    
- grounded QA 适合 Retrieval QA with Sources
    
- prompt 要明确格式和约束
    
- 用户请求应结构化增强
    
- 二步 prompt 的第二步要精确约束
    
- 敏感领域优先加显式 guardrail
    
- 用“只基于上下文，不知道就说不知道”降低幻觉
    
- agent 要显式声明函数签名
    
- multi-step agent 关键是 tool routing
    
- 模型选型看任务、成本、延迟、评估结果
    
- 暴露个人信息属于 privacy leak
    

## Assembling and Deploying Applications

- MLflow PyFunc 重点是 `load_context` 和 `predict`
    
- 模型服务权限治理依赖 Unity Catalog
    
- 顺序链的基础单位是 chain
    
- prompt + model 的基础组件是 LLM Chain
    
- 基础 RAG 离不开 embedding model
    
- 模型进入 Unity Catalog 要走注册流程
    
- RAG 部署顺序是文档处理 → embedding → 索引 → retriever → 注册 → 服务
    
- Vector Search 通过 similarity search 查询
    
- 外部 API 模型场景部署的是 wrapper/agent
    
- embeddings 存 Delta 后还要建 vector index
    
- 批量推理优先 AI Query
    

## Governance

- 敏感字段要遮盖
    
- prompt injection 要靠输入治理
    
- offensive content 要预过滤
    
- 优先使用合法、自有、授权清晰的数据源
    
- toxic language 大规模治理依赖自动化分类
    
- 无关字段用占位符遮盖而不是粗暴删除
    

## Evaluation and Monitoring

- 法律摘要优先看 factual consistency
    
- customer-facing 应用持续看 response time 和 user engagement
    
- MLflow 记录 latency、quality metrics、retrieval outputs
    
- inference logging 提供生产可观察性
    
- serving concurrency 和扩缩容影响成本
    
- inference tables / agent monitoring 跟踪成功率和错误趋势
    
- precision / recall 依赖 ground truth
    
- evaluation 在上线前，monitoring 在上线后
    

---

# 十、一句话总总结

这整套内容的核心可以概括为：

**先把业务需求设计正确，再把数据准备正确，再把应用开发清楚，再把系统装配上线，同时通过治理、评估与监控，让生成式 AI 应用在真实环境中长期保持准确、稳定、安全、合规且成本可控。**
