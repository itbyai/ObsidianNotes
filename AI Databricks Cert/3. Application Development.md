# Application Development

## 模块定位

Application Development 是生成式 AI 应用考试中的重点模块，覆盖范围较大，主要考察如何围绕业务需求设计应用能力、选择合适组件、构建安全约束、优化提示、选择模型与嵌入方案，以及支持多步骤代理执行。

这一部分的核心不是单纯会调用模型，而是能够把一个应用真正搭起来，既能取数、检索、生成，又能兼顾安全性、成本、延迟和可扩展性。

本模块主要包含这些能力：

- 为特定检索需求构建合适的数据提取工具
    
- 选择适合 RAG/生成式应用的链式组件或框架能力
    
- 理解 prompt 格式如何影响模型输出
    
- 识别聊天机器人中的安全与隐私问题
    
- 根据评估指标选择更合适的 chunking 方案
    
- 将用户请求转化为结构化上下文，增强回答质量
    
- 设计多步提示，使模型输出满足明确约束
    
- 为敏感领域添加 guardrail
    
- 使用 meta prompt 降低幻觉
    
- 为 agent 设计支持函数调用的提示模板
    
- 根据延迟和成本要求选择模型
    
- 针对长文档检索设计嵌入方案
    
- 选择适合微调模型的筛选标准
    
- 为多工具代理选择关键框架能力

# 1. 为数据检索需求构建合适的工具

## 典型场景

需要从一个包含 CSV 文件的 S3 bucket 中获取每日销售数据，对这些数据进行处理，然后送入 RAG workflow。

## 推荐方案

最合适的工具是：

**一个读取 S3、处理数据并写出优化格式的 PySpark 作业**

## 原因

这个选择更合适，主要因为：

- PySpark 可以很好地与 S3 集成
    
- 能处理大规模数据
    
- 数据转换效率高
    
- 适合生成下游处理更友好的输出格式
    

## 不合适的选项类型

以下方案通常不如 PySpark 合适：

- 只用 Python requests 的简单 HTTP fetcher
    
- 循环本地 CSV 的 bash 脚本
    
- TensorFlow 数据上传脚本
    

这些方案要么不适合云端大规模批量数据读取，要么不适合结构化 ETL 场景。

## 核心原则

**当需求是从对象存储中批量读取、处理并标准化结构化数据时，优先选择数据工程型工具，而不是临时脚本型方案。**

# 2. 为 RAG 聊天机器人选择合适的链式组件

## 典型场景

要构建一个 retrieval-augmented chatbot，它既需要检索内容，又能调用外部 API，并和 Databricks 交互获取数据。

## 推荐组件

更合适的链式组件是：

**[[Retrieval QA with Sources Chain]]**

## 原因

这个组件的关键优势在于：

- 同时包含 retrieval 和 LLM 层
    
- 能够引用来源
    
- 非常适合 grounded QA
    
- 更适合需要基于检索内容回答、并展示依据的场景
    

## 为什么不是其他组件

单纯的 LLM chain 或 sequential chain 不能像 retrieval QA with sources 一样，天然突出“带来源的检索问答”这个需求重点。

## 核心原则

**当应用既要基于外部知识检索回答，又强调答案可追溯时，优先考虑带 sources 的 retrieval QA 结构。**

# 3. Prompt 格式会直接改变输出形式与结果质量

## 典型问题

如果希望模型输出一份政策文件的**要点式摘要**，什么样的 prompt 更合适？

## 更优的 prompt 特征

更有效的 prompt 应该：

- 明确任务是 summary
    
- 指定输出格式
    
- 限制输出结构
    

例如：

- 总结这份政策文件
    
- 用五个 bullet points 输出
    
- 给出 TLDR
    

## 原因

更精确、更具体的 prompt 能够让模型：

- 更容易理解目标任务
    
- 更稳定地产生目标格式
    
- 输出更贴近业务需要
    

## 核心原则

**不要只告诉模型“做什么”，还要明确告诉它“怎么输出”。**

---

# 4. 聊天机器人包含过时个人信息时，属于隐私泄露问题

## 典型场景

聊天机器人的回答中包含了关于某位用户的过时个人数据。

## 问题归类

这类问题应归类为：

**Privacy Leak**

## 原因

只要响应中暴露了用户个人信息，尤其是过时的、未经确认的、可能不应披露的信息，就属于明显的隐私与安全风险。

## 不是重点的问题类型

虽然实际系统中也可能伴随不相关或回答质量问题，但这里最核心的问题是：

- 个人数据被暴露
    
- 用户隐私边界被突破
    

## 核心原则

**涉及个人数据暴露时，首先按隐私与安全风险处理，而不是只当成普通回答错误。**

---

# 5. 使用检索指标比较两种 chunking 方案

## 典型场景

比较两种 chunking 方案：

- 512 token fixed chunks
    
- section-based chunks
    

发现 section-based chunks 的 **Recall@K 更高**。

## 应该如何选择

应选择：

**section-based chunks**

## 原因

更高的 Recall@K 表明：

- 检索召回效果更好
    
- 与查询需求的对齐更强
    
- 更容易把真正相关的信息召回出来
    

## 核心原则

**chunking 方案不靠主观感觉决定，要看检索评估指标。若某方案 Recall@K 更高，说明它更适合该检索场景。**

---

# 6. 把用户请求增强成结构化上下文

## 典型场景

用户输入：

**“Show me Q3 expenses for project Phoenix.”**

## 更合适的提示增强方式

不要只把原始问题直接丢给模型，而应把请求增强成结构化上下文，例如包含：

- quarter = Q3
    
- project = Phoenix
    
- expenses = requested metric
    

或者整理成更清晰的上下文表达，再交给模型生成最终回答。

## 原因

将原始自然语言请求转成更结构化的上下文，有助于模型：

- 识别关键字段
    
- 更准确定位数据
    
- 生成 context-aware response
    
- 减少歧义
    

## 核心原则

**用户问题往往是自然语言，模型和工具执行更依赖结构化上下文。增强结构化信息能显著提升回答精度。**

---

# 7. 设计二步提示时，第二步要给出精确约束

## 典型场景

模型先生成一份 200-word summary，然后第二步需要把它压缩到 50 字以内。

## 最佳第二步提示

更好的第二步提示应明确写出：

**“Shorten the summary to no more than 50 words.”**

## 原因

这里的关键在于“no more than 50 words”这一精确约束。  
明确的硬性约束能帮助模型：

- 更好遵循输出长度要求
    
- 降低输出超长的风险
    
- 让压缩任务更稳定
    

## 核心原则

**在多步 prompt 中，第二步不能只说“缩短一下”，要明确给出具体约束。**

---

# 8. 在敏感领域通过条件拒绝实现 guardrail

## 典型场景

希望防止聊天机器人提供法律建议。

## 合适的 guardrail

更合适的方法是：

**在 prompt logic 中加入条件拒绝规则，例如当用户请求法律建议时拒绝回答。**

## 为什么这比微调更合适

虽然微调有时也是一种选择，但存在问题：

- 计算成本高
    
- 实施周期长
    
- 需要大量谨慎设计
    
- 不适合作为首选的快速安全措施
    

相比之下，条件拒绝逻辑：

- 更直接
    
- 更易控
    
- 更容易审查
    
- 更适合作为显式安全护栏
    

## 核心原则

**对于高风险领域的禁止行为，优先用明确规则构建 guardrail，而不是先指望模型自行“学会”拒绝。**

# 9. 用 meta prompt 降低 hallucination

## 典型问题

什么样的 meta prompt 最能减少幻觉回答？

## 推荐方式

更好的提示原则是：

- 只使用提供的上下文
    
- 如果不确定，就明确说 “I don’t know”
    

## 原因

这类提示能够鼓励模型：

- 保持谨慎
    
- 不随意补全缺失信息
    
- 不依赖模糊记忆或臆测
    
- 提高 factual fidelity
    

## 不推荐的方向

以下做法更容易增加幻觉：

- 根据记忆作答
    
- 估算缺失细节
    
- 依赖内部知识自行补全
    

## 核心原则

**降低 hallucination 的关键，不是让模型更大胆，而是让模型在不确定时学会收住。**

---

# 10. 为 agent 设计支持函数调用的提示模板

## 典型场景

要让 agent 能调用函数来获取数据。

## 更合适的 agent prompt 模板

最佳做法是：

**在提示中声明可用函数及其签名**

## 原因

当 agent 明确知道：

- 有哪些函数可用
    
- 每个函数的输入参数是什么
    
- 应该在什么场景调用这些函数
    

它的行为会更：

- 结构化
    
- 可控
    
- 准确
    

## 核心原则

**函数调用能力不是靠暗示，而是靠显式声明接口。**

---

# 11. 在低延迟、低成本实时应用中选择合适模型

## 典型场景

应用要求：

- real-time answers
    
- constrained latency
    
- low cost
    

## 推荐模型

更合适的选择是：

**GPT-3.5 Turbo**

## 原因

它在该类场景下的优势是：

- 速度较快
    
- 成本较低
    
- 对中等复杂度任务足够实用
    
- 比更大更贵的模型更适合实时、成本敏感场景
    

## 对比理解

- 更大的模型通常能力更强，但成本和延迟更高
    
- 本地 CPU 跑开源模型在很多场景下也未必能达到理想实时性和维护成本
    
- 因此在“够用且快”的业务目标下，轻量托管模型更占优
    

## 核心原则

**模型选型要看业务约束，低延迟和低成本优先时，选速度和成本更平衡的模型。**

# 12. 长文档检索时应采用分段嵌入，而不是整篇单一嵌入

## 典型场景

源文档长度约 10K tokens，而查询很短。

## 更优的嵌入方式

更合适的方法是：

**将文档切成大约 1536-token 的 spans，再进行嵌入**

## 原因

这种分段嵌入方式能够：

- 保留更细粒度的语义单位
    
- 让短查询更容易匹配到具体相关片段
    
- 避免整篇单一嵌入过于粗糙
    
- 在大文档场景中提升检索质量
    

## 核心原则

**长文档不要一篇只做一个 embedding，分段嵌入更适合精细检索。**

---

# 13. 选择微调摘要模型时，要同时看任务匹配与评估表现

## 典型场景

需要选择一个适合摘要任务、且预算可控的微调模型。

## 关键筛选标准

更重要的筛选维度包括：

- 是否适合 summarization task
    
- token limit 是否满足场景
    
- 成本是否在预算内
    
- 是否有相应的 metadata tags
    
- 是否有可靠的 performance metrics
    
- 是否有人工评估或任务相关评估结果
    

## 为什么不能只看热度

单纯看下载量、知名度或流行程度并不够，因为这些不一定代表：

- 适合你的任务
    
- 满足你的成本约束
    
- 在你的场景下效果好
    

## 核心原则

**选模型要看“任务匹配 + 成本约束 + 评估证据”，不能只看流行度。**

# 14. 多步骤代理的关键能力是工具路由

## 典型场景

要构建一个 multi-step agent，它需要：

- 查询数据库
    
- 执行 web search
    
- 发送 email
    

## 关键框架能力

这种场景最关键的能力是：

**Tool Routing**

## 原因

多步骤代理的核心不只是一次 LLM 调用，而是要让系统能够：

- 根据任务动态决定调用哪个工具
    
- 在多个工具之间链式执行
    
- 按步骤推进整个任务流
    

这种动态工具路由能力，是多工具代理框架中的核心能力之一。

## 核心原则

**真正的 agent 不只是“会回答”，而是“会决定下一步该用哪个工具”。**

# 15. 本模块的统一方法论

虽然 Application Development 题型很多，但背后的方法论非常一致。

## 15.1 应用开发关注的是“系统行为”

不是只看单个 prompt，而是看整个应用是否能：

- 取数
    
- 检索
    
- 生成
    
- 约束风险
    
- 路由工具
    
- 满足性能要求
    

---

## 15.2 结构化设计优于模糊调用

这一部分反复强调结构化思维：

- 数据获取要有合适工具
    
- 查询增强要结构化
    
- agent 调用函数要声明签名
    
- 多步任务要有工具路由
    
- 输出约束要明确写进 prompt
    

---

## 15.3 安全性和实用性同样重要

这部分不仅讲“怎么让模型更会答”，也在讲：

- 不该答的要拒绝
    
- 不确定时要说不知道
    
- 不应暴露个人信息
    
- 高风险领域要加 guardrail
    

说明 Application Development 不只是能力设计，也是安全设计。

---

## 15.4 模型和方案选择要服务业务约束

多个题目都在强调取舍：

- 低成本 vs 高能力
    
- 低延迟 vs 大模型
    
- 单一嵌入 vs 分段嵌入
    
- 微调 vs 显式规则
    
- 流行度 vs 真实任务适配
    

这说明考试很重视“工程判断力”。
