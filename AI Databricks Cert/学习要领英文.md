
# Databricks Generative AI Engineer Associate

## Cleaned Study Notes (Part 1)

## 1. Exam Overview

The material describes the certification as testing whether a candidate can design and implement **LLM-enabled applications on Databricks**, including choosing models, decomposing business problems, building RAG systems, deploying solutions, and managing them with Databricks-native tools such as **Vector Search, Model Serving, MLflow, and Unity Catalog**.

The guide says the exam is divided into six domains:

- Design Applications — 14%
    
- Data Preparation — 14%
    
- Application Development — 30%
    
- Assembling and Deploying Apps — 22%
    
- Governance — 8%
    
- Evaluation and Monitoring — 12%
    

This part of the guide mainly covers the first four domains.

# 2. Cleaned Core Content

## Section 1: Design Applications

### 1.1 Designing prompts for specific output formats

A strong prompt should include:

- a clear instruction
    
- enough context
    
- the actual input or question
    
- the desired output format
    

Important prompt design ideas mentioned in the source:

- use delimiters to separate instruction from context
    
- ask for structured output such as JSON
    
- use zero-shot, few-shot, or prompt chaining when needed
    
- explicitly tell the model not to fabricate answers if information is missing
    

In short:  
**Good prompts are explicit, contextual, and format-aware.**

---

### 1.2 Selecting model tasks for a business requirement

The right process is not “pick a model first.”  
Instead:

1. identify the business objective
    
2. decompose it into smaller tasks
    
3. map each task to model capabilities
    
4. decide the order and relationship between tasks
    
5. then choose the model(s)
    

Example from the material: customer service may involve sentiment analysis, FAQ retrieval, and response generation.

Core idea:  
**Business need → task breakdown → model/task alignment.**

---

### 1.3 Selecting chain components

The guide highlights common chain components such as:

- prompt
    
- retriever
    
- tool / function calling
    
- LLM
    

It also mentions frameworks like:

- LangChain
    
- LlamaIndex
    
- OpenAI Agents
    

The real takeaway is understanding how a workflow is assembled from these parts, not memorizing framework APIs.

---

### 1.4 Translating business goals into AI pipeline inputs and outputs

Candidates should be able to define:

- what goes into the pipeline
    
- what should come out
    
- what intermediate transformations are needed
    
- which model is best for each stage
    

Key concerns include:

- data quality
    
- model suitability
    
- integration across components
    
- testing and optimization
    

---

### 1.5 Ordering tools for multi-step reasoning

The guide emphasizes agent-style reasoning patterns such as:

- **ReAct**: Reason → Act → Observe
    
- tool use / function calling
    
- planning
    
- multi-agent collaboration
    

You should understand when a task is:

- single-step
    
- sequential
    
- graph-like and interdependent
    

And how tools should be ordered to support that reasoning flow.

## Section 2: Data Preparation

### 2.1 Chunking strategy

Chunking is presented as a major RAG concept.

Strategies mentioned include:

- sentence/paragraph/section-based chunking
    
- fixed-size token chunking
    
- windowed summarization or context-preserving chunking
    

Chunking should be chosen based on:

- model context window
    
- document structure
    
- retrieval precision vs context preservation
    

General principle:

- **smaller chunks** improve precision
    
- **larger chunks** preserve broader context
    

The right strategy depends on evaluation and use case.

---

### 2.2 Filtering irrelevant content from source documents

The guide stresses removing noise that weakens retrieval quality, such as:

- ads
    
- navigation bars
    
- footers
    
- irrelevant text
    
- formatting noise
    
- misspellings and inconsistent normalization
    

The purpose is to improve the quality of the text before embedding and retrieval.

---

### 2.3 Choosing Python tools for document extraction

The source mentions selecting extraction tools based on source format, including:

- PDF extraction libraries
    
- OCR tools when text is image-based
    
- tools appropriate for HTML, DOC, or other document types
    

The exam-relevant idea is:

- text PDFs can be parsed directly
    
- scanned/image-based files need OCR
    
- tool choice depends on the source format and extraction quality required
    

---

### 2.4 Writing chunked text to Delta Lake and Unity Catalog

The sequence described is roughly:

1. ingest the document text
    
2. clean it
    
3. chunk it
    
4. compute embeddings
    
5. store chunks and embeddings in Delta Lake
    
6. register and govern data through Unity Catalog
    
7. support downstream retrieval workflows
    

The material also mentions using automation and Delta Live Tables for refresh and orchestration.

---

### 2.5 Choosing source documents for RAG

Document selection should be based on:

- relevance
    
- quality
    
- accuracy
    
- reliability
    
- completeness
    
- diversity of coverage
    

The best knowledge base is not simply the largest one; it is the one most aligned with the task.

---

### 2.6 Identifying prompt/response pairs

The guide frames this as selecting examples aligned with the target task.

Important qualities:

- task alignment
    
- contextual relevance
    
- quality control
    
- correct tagging/labeling
    

Examples mentioned include sentiment labels and question-answering categories.

---

### 2.7 Evaluating retrieval performance

The material lists several retrieval-related metrics:

- Context Precision
    
- Context Recall
    
- Faithfulness
    
- Answer Relevancy
    
- Answer Correctness
    

It also mentions:

- MLflow for evaluation workflows
    
- LLM-as-a-Judge
    
- offline vs online evaluation
    
- task-specific metrics like BLEU and ROUGE
    

A key exam idea is that poor answer quality may come from retrieval issues, not just model issues.

## Section 3: Application Development

The source describes a typical RAG preparation pipeline:

- extract document content
    
- chunk it
    
- embed it
    
- store it in a vector system
    
- retrieve relevant context later
    

Challenges include:

- complex document structure
    
- preserving logical sections
    
- multimodal content with text and images
    

---

### 3.2 Selecting LangChain or similar frameworks

The guide points to frameworks that help orchestrate:

- prompt templates
    
- memory
    
- chains
    
- retrievers
    
- tools
    
- LLM calls
    

The focus is not on framework trivia, but on whether the framework supports your app architecture well.

---

### 3.3 Understanding how prompt format changes outputs

Prompt structure directly affects:

- quality
    
- hallucination risk
    
- response relevance
    
- formatting consistency
    

Prompt augmentation with retrieved context is also emphasized as a major improvement technique.

---

### 3.4 Qualitative assessment of responses

The guide says you should be able to spot:

- hallucinations
    
- bias
    
- incomplete answers
    
- unsafe outputs
    
- low-quality responses
    

This is not just about whether the answer “works,” but whether it is safe and reliable.

---

### 3.5 Choosing chunking strategy using evaluation results

Chunking should be treated as an optimization variable.

You may need to adjust chunk size and strategy depending on:

- model context window
    
- retrieval quality
    
- precision vs coverage trade-offs
    

This is meant to be evaluated iteratively, not fixed blindly.

---

### 3.6 Prompt augmentation

The source describes adding context to prompts based on:

- user terms
    
- key fields
    
- intent
    
- retrieved external knowledge
    

This is the core of RAG:

**user query + retrieved context + instruction template = improved answer quality**

---

### 3.7 Prompt refinement toward desired output

To move from a baseline response to a desired one, the guide suggests:

- clearer instructions
    
- better formatting constraints
    
- examples
    
- delimiters
    
- iterative tuning
    
- parameter adjustment such as temperature
    

This reflects practical prompt engineering.

---

### 3.8 Guardrails and metaprompts

Two important safety/control concepts appear:

**Guardrails**

- prevent harmful or inappropriate outputs
    
- can be simple prompt-based rules
    
- can also use dedicated safety models
    

**Metaprompts**

- reduce hallucinations
    
- reduce private data leakage
    
- provide higher-level behavioral guidance
    

Simple summary:  
**Prompts guide the model; guardrails constrain it.**

The source highlights prompt templates that tell the model:

- what tools are available
    
- what each tool does
    
- how functions should be called
    

This is important in agent-style application design.

- least privilege principles

### 4.3 Coding simple chains

A simple chain is described as a structured sequence of steps where each step transforms input and passes output onward.

Typical elements:

- prompt
    
- model
    
- optional retriever
    
- optional tool usage
    
- output transformation
    

LangChain is used as an example framework for this.

The guide lists core RAG building blocks such as:

- model flavor
    
- embedding model
    
- retriever
    
- dependencies
    
- input examples
    
- model signature

This is one of the most practical parts of the material.

### 4.5 Registering models in Unity Catalog with MLflow

The source explains the idea of:

- logging the model
    
- registering it in Unity Catalog
    
- managing versions
    
- supporting lifecycle tracking and lineage
    
- loading it later for inference
    

Core idea:  
**MLflow + Unity Catalog = governed model lifecycle management.**

### 4.6 Sequencing RAG deployment steps

A simplified deployment sequence inferred from the material is:

1. prepare source documents
    
2. extract and clean text
    
3. chunk the content
    
4. generate embeddings

### 4.7 Creating and querying a Vector Search index

The guide says a Vector Search index can be built by syncing with embedding data stored in Delta tables.

Important ideas:

- supports approximate nearest neighbor search
    
- can be queried through REST API or Python SDK
    
- supports syncing and CRUD
    
- integrates with Unity Catalog governance

# 4. The Real High-Value Exam Themes

If you want the shortest useful version, remember these ten ideas:

1. A good prompt has instruction, context, input, and output format.
    
2. Business requirements should be decomposed into model tasks.
    
3. RAG pipeline basics: extract → clean → chunk → embed → retrieve → augment prompt → generate.
    
4. Chunking strongly affects retrieval quality.
    
5. Source document cleaning matters.
    
6. Retrieval should be evaluated with metrics like precision, recall, faithfulness, relevancy, and correctness.
    
7. Guardrails and metaprompts help reduce harmful output, hallucinations, and leakage.
    
8. Core RAG pieces include embedding model, retriever, vector index, LLM, dependencies, and model signature.
    
9. MLflow and Unity Catalog are central for model lifecycle and governance.
    
10. Databricks deployment knowledge includes Vector Search, Model Serving, Foundation Model APIs, access control, and monitoring.
