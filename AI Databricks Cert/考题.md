核心不是死记答案，而是看你能不能抓住 **题干里的“关键词”**。这 10 题主要考 5 类能力：**合规意识、LLM 编排、RAG 流程、Databricks 在线服务/监控、向量检索匹配设计**。LangChain 官方文档明确覆盖 retrieval、tools 和 agent workflow；Databricks 官方文档则把 RAG、Inference Tables、Feature Serving、Vector Search 的定位讲得很清楚。([LangChain Docs](https://docs.langchain.com/oss/python/langchain/overview?utm_source=chatgpt.com "LangChain overview - Docs by LangChain"))

## Q73

**答案：B**

这题考的是 **RAG 标准流程顺序**。  
合理顺序应该是：

**Collect documents → Index/store → Accept query → Retrieve chunks → Generate response → Evaluate → Deploy**

Databricks 官方把 RAG 解释为 retrieval + generation 的应用范式，同时也有专门的评估与改进文档，说明评估是构建流程里很重要的一环。([Databricks Documentation](https://docs.databricks.com/aws/en/generative-ai/retrieval-augmented-generation?utm_source=chatgpt.com "RAG (Retrieval Augmented Generation) on Databricks"))

为什么是 B：

- 先建知识索引
    
- 再接收问题
    
- 再检索相关 chunk
    
- 再生成答案
    
- 再评估
    
- 最后部署
    

为什么不是其他：

- **A**：把 evaluate 放在 generate 前面，不顺。
    
- **C**：漏掉了 query / retrieval / generation。
    
- **D**：还没收集建索引，就先 accept user queries，不符合构建逻辑。
    

这题最容易错在：  
把 **evaluation** 放到奇怪的位置。考试里通常默认是 **先把应用跑起来，再评估，再上线/迭代**。

## Q74

**答案：B. Choose a smaller LLM that is tuned for IT domain**

题干有两个核心限制：

- **cost-conscious**
    
- **internal infrastructure documentation**
    

所以最优思路不是“上最大模型”，而是 **选更小但更贴近 IT 领域的模型**。  
因为这是 RAG，不是纯模型裸答，效果很大程度来自 **检索质量 + 领域匹配**。

为什么不是其他：

- A 限制文档数量，会伤害召回
    
- C 限制提问次数，只是限流，不是提高质量
    
- D 用最大模型，和 cost-conscious 直接冲突
    

这题考的是一个很典型的产品思维：  
**不是最强模型，而是最合适、最省钱、够用的方案。**

## Q76

## Q77

**答案：D. Inference Tables**

这题是 Databricks 里面非常标准的一题。  
题干说现在用自建 micro-service 把请求/响应日志发到远程服务器，现在想换成 Databricks 原生能力。Databricks 官方文档对 inference tables 的定义就是：**自动捕获 serving endpoint 的 incoming requests 和 outgoing responses，并记录到 Unity Catalog Delta table**。这和题干一一对应。([Databricks Documentation](https://docs.databricks.com/aws/en/ai-gateway/inference-tables?utm_source=chatgpt.com "Monitor served models using AI Gateway-enabled ..."))

为什么不是其他：

- Vector Search：做检索
    
- Lakeview：做展示/分析视图
    
- DBSQL：做查询
    
- **Inference Tables：做在线推理输入输出日志**
    

补一个现在容易混的点：  
Databricks 新文档里已经更强调 **AI Gateway-enabled inference tables**，而老的 inference tables 正在退出，但考试语境下，这题的正确概念仍然是 **Inference Tables**。([Databricks Documentation](https://docs.databricks.com/aws/en/ai-gateway/inference-tables?utm_source=chatgpt.com "Monitor served models using AI Gateway-enabled ..."))

## Q78

**答案：C. Feature Serving**

题干强调：

- **real-time updates**
    
- 最新 scores 和 events
    
- 要把这些新鲜数据送进 LLM 生成 live summary
    

Databricks 官方文档说明，Feature Serving 用来给实时应用提供低延迟在线数据/特征；Online Feature Store 文档也明确说它面向 real-time applications。这个定位和题意完全一致。([Databricks Documentation](https://docs.databricks.com/aws/en/machine-learning/feature-store/feature-function-serving?utm_source=chatgpt.com "Feature Serving endpoints | Databricks on AWS"))

为什么不是其他：

- Foundation Model APIs：负责模型调用，不是实时数据供给层
    
- AutoML：自动训练/选模，不是在线数据注入
    
- DatabricksIQ：不是这里考的“新鲜数据输入”能力
    

这题的记忆法：  
**模型负责推理，Feature Serving 负责把最新结构化数据及时送到模型前面。**

## Q79

**答案：D**

这题很典型，考的是 **structured filter + semantic matching**。

题目要求两件事同时满足：

1. 员工 **日期可用**
    
2. 员工 profile 和 project scope **语义匹配**
    

Databricks Vector Search 官方文档说明，向量索引可以包含 embedded data with metadata，并且支持 filtering，还专门提到适合 real-time matching applications。对这种场景，最合理的设计就是：

- 把 **employee profiles** 做 embedding 存进向量库
    
- 用 **availability** 做过滤
    
- 用 **project scope** 作为查询去检索最匹配员工
    

这正是 D。([Databricks Documentation](https://docs.databricks.com/aws/en/vector-search/vector-search?utm_source=chatgpt.com "Mosaic AI Vector Search | Databricks on AWS"))

为什么不是其他：

- A：把 project scope 存进库，再用 employee profile 去检索，方向反了
    
- B：keyword extraction 太脆弱，语义损失大
    
- C：可以做，但不像题目想考的可扩展向量检索架构
    

这题一眼看法：  
**时间可用性 = filter，文本匹配 = vector similarity**。
