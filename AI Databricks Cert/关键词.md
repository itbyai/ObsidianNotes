## Foundations / 平台基础

- Databricks Workspace — Databricks 的工作空间界面，用来管理 notebook、作业、模型和应用。
    
- Compute — 运行 notebook、任务和数据处理作业的计算资源。
    
- Cluster — 传统交互式或作业型计算集群。
    
- Serverless Compute — 由平台托管、免运维的计算模式。
    
- SQL Warehouse — 面向 SQL 查询和 BI 工作负载的计算层。
    
- Playground — 用来快速试 prompt、测试模型行为的交互式界面。
    
- Unity Catalog — Databricks 的统一治理层，用来管理数据、模型、函数与权限。
    
- Catalog — Unity Catalog 的顶层命名空间。
    
- Schema — Catalog 下的逻辑分组，用来组织表、视图、模型和函数。
    
- Table — 存储结构化数据的对象。
    
- View — 基于查询定义的逻辑对象，本身通常不存底层数据。
    
- Foundation Model API — 通过 API 方式调用托管基础模型的能力。
    

## Design Applications / 应用设计

- Prompt Engineering — 设计提示词以稳定、准确地驱动模型输出。
    
- Zero-shot Prompting — 不给示例，直接让模型完成任务。
    
- Few-shot Prompting — 在提示中给少量示例来约束模型行为。
    
- Chain-of-thought Prompting — 引导模型分步骤推理的提示方式。
    
- Structured Output — 要求模型输出 JSON、字段化文本或固定模板。
    
- System Prompt — 设定模型角色、边界和总体行为的高优先级提示。
    
- Temperature — 控制输出随机性；越低越稳定，越高越发散。
    
- Model Task Selection — 根据业务需求选择最合适的模型任务类型。
    
- Summarization — 将长文本压缩成较短总结。
    
- Classification — 把文本归入预定义类别。
    
- Question Answering — 根据问题和上下文生成答案。
    
- Feature Extraction — 从文本中抽取可用于下游系统的结构化特征。
    
- Token Classification — 对每个 token 做分类，如实体识别或词性标注。
    
- BoW — Bag of Words，基于词频的经典文本表示法。
    
- TF-IDF — 用词频与逆文档频率衡量词的重要性。
    
- Embeddings — 用稠密向量表示文本语义相似性。
    
- NER — Named Entity Recognition，识别人名、地点、组织等实体。
    
- POS Tagging — Part-of-Speech Tagging，给词标注词性。
    
- Chain Design — 设计从输入到输出的链式处理流程。
    
- Retriever → Prompt → LLM → Parser — 典型链式架构：先检索，再组装提示，再生成，再解析结果。
    
- ReAct — 结合推理与行动的 agent 模式。
    
- Router Chain — 根据意图把请求路由到不同子链或工具。
    
- Agent Bricks — Databricks 提供的可复用 agent 设计模式。
    
- Knowledge Assistant — 面向知识检索与问答的 agent 模式。
    
- Multiagent Supervisor — 由一个协调者管理多个子 agent 的模式。
    
- Information Extraction Brick — 从文档中提取结构化字段的 agent/流程模式。
    

## Data Preparation / 数据准备

- Document Extraction — 从原始文档中提取可用文本内容。
    
- pytesseract — 针对图片或扫描件做 OCR 文本提取的工具。
    
- pypdf — 从文本型 PDF 中解析文本的工具。
    
- beautifulsoup4 — 从 HTML/网页中抽取正文内容的工具。
    
- Header/Footer Filtering — 去掉页眉页脚等重复噪声。
    
- Feature Store — 存储结构化特征，用于结构化检索或补充上下文。
    
- Chunking — 把长文档切成适合检索和模型输入的小块。
    
- Fixed-size Chunking — 按固定长度切块，简单但可能破坏语义边界。
    
- Structure-aware Chunking — 按标题、段落、表格等结构切块。
    
- Token-based Chunking — 按 token 数量切块，更贴近模型上下文限制。
    
- Hierarchical Chunking — 分层切块，如 parent-child 结构。
    
- Parent-child Retrieval — 先检索子块，再回溯到父块获取更完整上下文。
    
- Chunk Size — 每个块的长度，会影响 recall、precision 和上下文完整性。
    
- Chunk Overlap — 相邻块之间的重叠区域，用于减少语义断裂。
    
- Delta Lake Pipeline — 用 Delta Lake 组织抽取、清洗、切块与入库流程。
    
- Change Data Feed (CDF) — 捕获表级增量变化，支持索引同步。
    
- Delta Sync Index — 由 Delta 表自动同步到 Vector Search 的索引方式。
    
- Direct Vector Access — 手动写入向量和文档的索引方式。
    
- Retrieval Evaluation — 评估检索器是否找到了足够相关的上下文。
    
- Precision@K — 前 K 个检索结果里相关项所占比例。
    
- Recall@K — 所有相关项中被前 K 个结果找回的比例。
    
- MRR — Mean Reciprocal Rank，关注第一个正确结果出现得有多靠前。
    
- MAP@K — Mean Average Precision at K，综合考虑多个相关项的排序质量。
    
- nDCG — 归一化折损累计增益，强调高位相关结果更重要。
    
- Context Relevancy — 检索上下文与问题是否真正相关。
    
- Context Sufficiency — 检索上下文是否足够支持正确回答。
    
- Re-ranking — 对初始召回结果再排序，提升 Top K 质量。
    
- Cross-encoder Re-ranker — 用更强的相关性模型对 query-document 对重新打分。
    
- Two-stage Retrieval — 先粗召回，再精排的检索架构。

## Governance / 治理

- PII — Personally Identifiable Information，个人可识别信息。
    
- Static Preprocessing — 在进入模型前先做脱敏或过滤，通常更稳妥。
    
- Inference-time Instructions — 只靠提示词要求模型不要泄露敏感信息。
    
- Full Exclusion — 直接把高风险内容从语料或上下文中排除。
    
- Data Licensing — 数据或模型能否被合法使用、修改和再分发。
    
- Apache 2.0 — 宽松开源许可证，允许商业使用和修改。
    
- MIT License — 更简洁宽松的开源许可证。
    
- CC BY — 允许使用但要求署名。
    
- CC BY-NC — 允许使用但不允许商业用途。
    
- CC BY-SA — 允许改编但要求相同方式共享。
    
- Proprietary License — 专有许可，使用和传播通常受更严格限制。
    
- GDPR — 欧盟数据保护法规，强调个人数据处理的合法性与最小化。

## 常见考试表达 / 容易混淆的概念

- Similarity Search — 按语义相似度查找最相近文档或向量。
    
- Semantic Search — 基于语义而非纯关键词的检索。
    
- Recall-oriented Retrieval — 偏向先尽量多召回，再做精排。
    
- Precision-oriented Retrieval — 偏向提高前几条结果的相关性纯度。
    
- Hallucination — 模型生成了看似合理但缺乏依据的内容。
    
- Guardrail — 对输入、输出、工具和策略加约束，减少风险和越权。
    
- Grounded Answer — 有检索上下文或证据支撑的回答。
    
- Citation / Source Attribution — 把答案对应到支持它的来源文档或片段。
    
- Offline Enrichment — 离线批量补充标签、摘要、分类等属性。
    
- Online Inference — 面向实时请求的推理。
    
- Monitoring Drift — 观察上线后输入分布、输出质量或行为是否偏移。
    
- Reference Answer — 评估时作为参照的标准答案。
    
- Golden Set vs Ground Truth — Golden set 是精选测试集；ground truth 是单条样本的真实答案/标签。
    
- Retriever Quality vs Generator Quality — 前者看找得是否对，后者看答得是否对。
    
- Evaluation before Deployment — 上线前做系统性质量验证。
    
- Monitoring after Deployment — 上线后持续看质量、延迟、错误和使用量。([Databricks](https://www.databricks.com/learn/certification/genai-engineer-associate "Databricks Certified Generative AI Engineer Associate | Databricks"))
